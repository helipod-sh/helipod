You've got a `stackbase serve` deployment running (see [Self-hosting](/docs/deploy/self-hosting)). Now
you want to ship a code change to it without rebuilding the container or restarting the process.

There are two ways to do that, plus a third path if you'd rather skip containers entirely:

- `stackbase deploy` pushes local `stackbase/` changes to an already-running `serve` and hot-swaps them in
  live, with no restart.
- `stackbase build` compiles your whole app into one self-contained executable, for when you'd rather
  ship a binary than a Docker image.
- The provisioning targets (`cloudflare`, `docker`, `railway`, `fly`, `aws`), a different
  `stackbase deploy` mode entirely. They provision infrastructure, they don't hot-swap.

This page covers all three. If you already know what you're looking for, jump straight there:

- [`stackbase deploy`: live hot-swap](#stackbase-deploy-live-hot-swap-onto-a-running-server)
- [The six deploy targets](#the-six-deploy-targets)
- [Deploying from CI (GitHub Actions)](#deploying-from-ci-github-actions)
- [`stackbase build`: one self-contained binary](#stackbase-build-one-self-contained-binary)
- [Choosing between `deploy`, `build`, and Docker](#choosing-between-deploy-build-and-docker-self-hosting),
  the comparison table, if you're not sure which of the three fits

## `stackbase deploy`: live hot-swap onto a running server

`stackbase deploy` transpiles your local `stackbase/` functions and additive schema changes, then pushes
them to a running `stackbase serve` deployment. The server validates everything and applies it
atomically: it keeps serving requests the whole time, and a rejected deploy never leaves it
half-updated.

<Steps>

<Step>

### Enable it on the server: `--allow-deploy`

A running `serve` doesn't accept deploys by default. Start it with `--allow-deploy`, or set
`STACKBASE_ALLOW_DEPLOY=1`, to opt in:

```bash
stackbase serve --dir stackbase --allow-deploy
# or, equivalently:
STACKBASE_ALLOW_DEPLOY=1 stackbase serve --dir stackbase
```

Without this flag, `POST /_admin/deploy` isn't registered at all. The request falls through to the
generic admin-router 404, indistinguishable from a nonexistent endpoint.

<Callout type="warn" title="Why this isn't on by default">

`STACKBASE_ADMIN_KEY` already grants full read and write access to your data (the dashboard and
`stackbase deploy` both authenticate with it). Without a separate opt-in, a leaked admin key would mean
remote code execution: every deploy replaces the functions that run *inside* the transaction. Requiring
`--allow-deploy` keeps "read/write my data" and "replace my running code" as two distinct, deliberately
opted-into capabilities, even though today they share one key.

</Callout>

</Step>

<Step>

### Push a deploy

From your app's project root (where `stackbase/` lives):

```bash
STACKBASE_ADMIN_KEY=your-strong-secret stackbase deploy --url https://myapp.example
```

| Flag | Env fallback | Default | Description |
|---|---|---|---|
| `--url <url>` | `STACKBASE_DEPLOY_URL` | (required, unless set in the config's `deploy` block) | The target deployment's base URL |
| `--dir <path>` | none | `stackbase` | The local `stackbase/` directory to push |
| `--check` | none | off | Fail if committed `stackbase/_generated/` has drifted from a fresh codegen run |
| `--dry-run` | none | off | Validate (preflight + package) without pushing |

`STACKBASE_ADMIN_KEY` is read from the environment (there's no `--admin-key` flag), and it's required.
`stackbase deploy` refuses to run without it. The general `--target`/`--env` flags also apply here
(this command form is the default `--target serve --env production`); see
[the CLI reference](/docs/reference/cli#stackbase-deploy).

<Callout type="warn" title="Run codegen yourself before a serve-target deploy">

The `serve` target pushes your tree exactly as it sits on disk. It does not regenerate
`stackbase/_generated/`. If your schema or functions changed, run `stackbase codegen` (and commit the
result) before deploying. `--check` verifies exactly this and fails fast on drift.

</Callout>

On success, verbatim:

```text
✓ deployed via serve (production) — rev 4b3187d88b93 (7 functions, 2 changed)
  https://myapp.example
```

`rev` is a content hash of the pushed file tree (the first 12 hex characters of its SHA-256), not a
sequence number. Deploying the exact same tree twice produces the same `rev`. The `2 changed` part
appears on a delta push (see
[what a deploy actually does](#going-deeper-what-a-deploy-actually-does) below); a full push prints
just `(<n> functions)`. The new functions are callable immediately, with no restart, and existing
WebSocket subscriptions keep working. They reactively pick up writes made through the newly deployed
code, exactly as if the change had been made via `stackbase dev`'s hot reload.

</Step>

</Steps>

### Responses and status codes

| Condition | HTTP status | CLI behavior |
|---|---|---|
| `--allow-deploy` not set on the target | 404 | `✗ deploy failed: deploy not enabled on target (start serve with --allow-deploy)`, exit 1 |
| Wrong or missing admin key | 401 | `✗ deploy failed: unauthorized`, exit 1 (checked before any file is written) |
| Malformed payload (bad JSON, unsafe path) | 400 | `✗ deploy failed: <reason>`, exit 1 |
| Destructive schema change | 409 | `✗ deploy failed: <reason>`, exit 1 (old version stays fully live) |
| Success | 200 | the two-line `✓ deployed via serve …` output shown above, exit 0 |

### Additive-only schema: destructive changes are rejected, not migrated

There are no data migrations in Stackbase today. `diffSchema` compares your pushed schema against the
server's live one, field by field and table by table.

**Allowed** (accepted, swap proceeds):
- Adding a brand-new table.
- Adding a new optional field to an existing table.

**Rejected** (409, server stays on the previous version):
- Removing or renaming a table.
- Changing a table's internal table number. This shouldn't happen under normal use: table numbers are
  stable across a deploy specifically so this check never fires on a legitimate change.
- Removing a field from an existing table.
- Changing a field's type, including what looks like a safe widening, like `string` to a union that
  includes `string`, or `any` to `string`. The diff can't verify a "widening" is actually sound against
  every existing row (an `any` column may hold values that aren't valid `string`s), so v1 takes the
  conservative position: any field-type change is rejected. Over-rejecting only fails a deploy. It can
  never corrupt data, which is the property this gate exists to protect.
- Turning an existing optional field required.
- Adding a new required field to an existing table. Existing rows don't have it, so make it optional
  instead, and backfill via a mutation if you need every row populated.

Any of these come back as `deploy failed: <reason>` describing exactly which table and field failed and
why:

```
✗ deploy failed: field "notes.priority" changed type string→number (destructive)
```

### The component set is fixed at boot

Components composed via `stackbase.config.ts` (`@stackbase/scheduler`, `@stackbase/workflow`, and any
others) are whatever was declared when `serve` booted. `stackbase deploy` can push new functions and
additive schema against that fixed component set, but it can't add or remove a component on a live
server. Changing the component list in `stackbase.config.ts` requires restarting the process, or
redeploying the container.

<Callout type="warn" title="Functions must be top-level under stackbase/">

The function loader (used by `dev`, `serve`, and `deploy` alike) is top-level only. It does not recurse
into subdirectories. Keep your query, mutation, and action modules directly under `stackbase/*.ts` (for
example `stackbase/items.ts`), not nested under a subfolder like `stackbase/lib/items.ts`. Nested modules are
silently not loaded, in a deploy exactly as in local dev.

</Callout>

### Worked example

```bash title="terminal"
# 1. Start a deploy-enabled server (once)
STACKBASE_ADMIN_KEY=prod-secret stackbase serve --dir stackbase --allow-deploy &

# 2. Make a change locally: add a new query, add an optional field to schema.ts
#    (edit stackbase/notes.ts, stackbase/schema.ts)

# 3. Push it live (run `stackbase codegen` first if schema/functions changed)
STACKBASE_ADMIN_KEY=prod-secret stackbase deploy --url http://localhost:3000
# ✓ deployed via serve (production) — rev 4b3187d88b93 (8 functions, 2 changed)
#   http://localhost:3000

# 4. The new function is callable immediately, no restart:
curl -X POST http://localhost:3000/api/run \
  -H 'content-type: application/json' \
  -d '{"path":"notes:add","args":{"box":"b1","text":"hello"}}'
```

Any WebSocket client subscribed to a query touching the `notes` table before step 3 sees the write from
step 4 pushed to it reactively. The deploy in between didn't drop or reset the subscription.

### Going deeper: what a deploy actually does

<Accordions type="single">

<Accordion title="What happens on your machine">

`stackbase deploy` runs three steps locally, then hands off to one remote step:

1. **Transpiles every `.ts` file under `stackbase/`** via esbuild's `transform` (not `bundle`). This strips
   TypeScript types but leaves import specifiers completely untouched. A bare import like
   `import { v } from "@stackbase/values"` passes through as-is, and it resolves against the target
   server's own `node_modules`, not anything on your machine. Relative imports resolve within the
   pushed file tree itself. This is why a deploy target needs the same `@stackbase/*` packages
   installed as your app does (the Docker image already provides this, see
   [Self-hosting](/docs/deploy/self-hosting)). It does **not** refresh `stackbase/_generated/`; that's
   `stackbase codegen`'s job, run it yourself before deploying.
2. **Probes `GET /_admin/deploy/modules`** for the server's current per-module content hashes
   (lowercase-hex SHA-256 per pushed path). An older server, or one with deploy disabled, doesn't
   answer; the CLI just falls back to a full-tree push.
3. **Posts a delta** to `/_admin/deploy`, with `Authorization: Bearer $STACKBASE_ADMIN_KEY`: modules
   whose hash differs go as `changed` (`{path, code}`), matching ones as `unchanged` (`{path,
   sha256}`), so an unchanged module never crosses the wire twice. Without a probe base it posts the
   full `{files: [{path, code}]}` tree instead, and if the server reports the delta's base is stale
   (another deploy landed in between), the CLI retries once with the full tree.
4. **The server validates and applies it** (see the next section). This is the only step that runs
   remotely.

</Accordion>

<Accordion title="What happens on the server">

On the server, `applyDeploy` does the following, in this exact order:

1. **Writes the pushed tree** to disk under a per-revision directory (`<deployRoot>/<rev>/functions/`),
   rejecting any path that's absolute or contains a `..` segment (a path-traversal guard on the
   payload).
2. **Loads and pushes it** through the same `loadFunctionsDir → push` pipeline used at boot, against the
   deployment's fixed, already-composed component set and its current table numbers. Existing tables,
   app and component alike, keep the numbers they already have. Only genuinely new tables get fresh
   ones.
3. **Diffs the schema** (see the additive-only rules above). If a table or field change is destructive,
   the deploy is rejected here, and nothing has been swapped yet.
4. **Only if both of those succeed**, it atomically swaps four things in sequence, with no `await`
   between them: `runtime.setModules(...)`, `runtime.setTableNumbers(...)`, `setRoutes(...)` (for
   `httpAction` routes), and the admin API's live schema (`adminApi.setSchema(...)`). Because there's no
   `await` in this sequence, no request can observe a state where only some of these have updated.

A load error or a schema rejection both leave the running deployment completely untouched. The previous
version keeps serving every request throughout. That's what "atomic" means here: not that the swap is
instantaneous (it is), but that validation is guaranteed to fully complete, with zero mutation of
running state, before the first field is swapped.

</Accordion>

</Accordions>

## The six deploy targets

`stackbase deploy --url <url>` (documented above) is shorthand for one specific target:
`--target serve`, the default. The general form is `--target <name> --env <name>` (plus `--dry-run`
and `--check`), and the other five targets provision infrastructure instead of hot-swapping an
already-running process. Target settings and credentials can live in the `deploy` block of
`stackbase.config.ts` (see [Configuration](/docs/reference/configuration#the-deploy-block)).

| Target | What a push does | Runs codegen? | Notes |
|---|---|---|---|
| `serve` (default) | Pushes `stackbase/` to a running `serve --allow-deploy`, live hot-swap | No, run `stackbase codegen` yourself | This page, above |
| `cloudflare` | Reconciles `wrangler.jsonc` bindings, then `wrangler deploy` | Yes | [Cloudflare](/docs/deploy/cloudflare); needs `wrangler` installed, `CLOUDFLARE_API_TOKEN` in CI |
| `docker` | `docker compose up -d --build` in your project | Yes | Needs Docker installed with the daemon running |
| `railway` | `railway up` (Railway builds the image itself) | Yes | Needs the `railway` CLI; `RAILWAY_TOKEN` in CI; optional `service`/`environment` settings |
| `fly` | `fly deploy` (Fly builds the image itself) | Yes | Needs `flyctl`; `FLY_API_TOKEN` in CI; optional `app`/`region` settings |
| `aws` | `aws apprunner start-deployment` against an existing App Runner service | Yes | Needs the `aws` CLI and a `serviceArn` setting; `AWS_ACCESS_KEY_ID` or `AWS_PROFILE` in CI |

Provider CLIs are never bundled into stackbase; each target shells out to the one you already have
installed, and fails `preflight` with a clear install hint if it's missing. Only the `serve` target
uses the additive-schema gate described above. The provisioning targets ship a whole new
build/image, so there's nothing to gate: the platform's own rollout replaces the process.

## Deploying from CI (GitHub Actions)

CI is just another caller of the same command: the workflow below runs the identical
`stackbase deploy` a human runs locally, with credentials coming from repository secrets.

```yaml title=".github/workflows/deploy.yml"
name: deploy
on:
  pull_request:
  push: { branches: [main] }
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: bun install
      - run: bun run test
      # PRs: verify committed codegen is current; exits before any target preflight, so
      # it needs no secrets. Add --dry-run to also validate target preflight/packaging
      # (the serve target's preflight then wants STACKBASE_DEPLOY_URL + STACKBASE_ADMIN_KEY).
      - if: github.event_name == 'pull_request'
        run: bunx stackbase deploy --check
      # main: the real deploy (serve target).
      - if: github.ref == 'refs/heads/main'
        run: bunx stackbase deploy
        env:
          STACKBASE_DEPLOY_URL: ${{ secrets.STACKBASE_DEPLOY_URL }}
          STACKBASE_ADMIN_KEY: ${{ secrets.STACKBASE_ADMIN_KEY }}
```

Three properties make this workflow safe to copy:

- **It never hangs.** `stackbase deploy` derives its interactive mode from `stdin.isTTY` and the
  `CI` environment variable (GitHub Actions sets `CI=true` on every runner), so a missing credential
  fails fast with an actionable message and a non-zero exit instead of prompting.
- **`--check` gates codegen drift.** It exits 1 if `stackbase/_generated/` doesn't match a fresh
  codegen run, which matters doubly for the `serve` target since a serve deploy never regenerates
  it. `--dry-run` additionally runs the target's `preflight` and `package` steps without pushing.
- **Exit codes are real.** A rejected schema change, a failed `--check`, or a provider CLI error all
  exit 1 and fail the job like any other CI step.

For a provisioning target, swap the deploy line and secrets: `bunx stackbase deploy --target
cloudflare` with `CLOUDFLARE_API_TOKEN` (plus a `npm i -D wrangler` install step), `--target
railway` with `RAILWAY_TOKEN`, `--target fly` with `FLY_API_TOKEN`, and so on per the table above.
Tokens always come from CI secrets; the provider CLIs' interactive `login` flows have no place in a
headless runner.

## `stackbase build`: one self-contained binary

`stackbase build` compiles your entire app, the Bun runtime, the Stackbase engine, `bun:sqlite`, your
`stackbase/` functions and schema, and any composed components, into a single executable via
`bun build --compile`. At deployment time you need nothing but the binary and a data directory: no
`bun`/`node` install, no `node_modules`, no `stackbase/` source tree.

| Component | Bundled into the binary? |
|---|---|
| Bun runtime | Yes |
| Stackbase engine (query engine, transactor, sync protocol) | Yes |
| SQLite (`bun:sqlite`) | Yes, built into Bun |
| Your `stackbase/` functions and schema | Yes, embedded via a generated static-import entrypoint |
| Composed components (`@stackbase/scheduler`, `@stackbase/workflow`, …) | Yes, whatever `stackbase.config.ts` composes |
| Dashboard | Optional, included by default (`--no-dashboard` to exclude) |
| SQLite database file | No, it lives on disk under `--data-dir`, external to the binary |

<Steps>

<Step>

### Check prerequisites

You need a `stackbase/` directory (schema and functions) and, optionally, a `stackbase.config.ts` next to
it: the same layout `stackbase dev` uses. There's no `stackbase init` scaffolder. If your app composes
components, install them as ordinary dependencies (`bun add @stackbase/scheduler @stackbase/workflow`).
`stackbase build` runs its own codegen internally, so there's no separate `stackbase codegen` step to
run first.

</Step>

<Step>

### Build

```bash
stackbase build
```

Output: `./stackbase-server` (the default `--outfile`).

```bash
stackbase build --outfile ./dist/my-backend
```

| Flag | Default | Description |
|---|---|---|
| `--dir <path>` | `stackbase` | App's `stackbase/` directory to build |
| `--outfile <path>` | `./stackbase-server` | Output path |
| `--target <platform>` | current platform | One of `linux-x64`, `linux-arm64`, `darwin-x64`, `darwin-arm64`, `windows-x64` |
| `--no-dashboard` | dashboard included | Exclude the dashboard UI from the binary |
| `--verbose` | off | Stream the underlying `bun build --compile` output instead of suppressing it |

On success, `stackbase build` prints the output path and its size:

```
✓ built ./stackbase-server (58MB)
```

</Step>

<Step>

### Run it

```bash
STACKBASE_ADMIN_KEY=your-strong-secret ./stackbase-server --port 3000 --hostname 0.0.0.0 --data-dir ./data
```

`STACKBASE_ADMIN_KEY` is required, exactly like `stackbase serve`. The binary fails fast (exit 1) if
it isn't set, and it's never auto-generated. This is production-facing, same as `serve`.

| Flag | Env fallback | Default | Description |
|---|---|---|---|
| `--port` | `PORT` | `3000` | Port to listen on |
| `--hostname` | none | `0.0.0.0` | Address to bind to |
| `--data-dir` | none | `./data` | Directory holding the SQLite database (`db.sqlite`) |
| `--database-url` | `STACKBASE_DATABASE_URL` | SQLite | Postgres connection string, same flag `serve` accepts. Unset uses the embedded SQLite adapter |

The binary is otherwise runtime-flag-compatible with `stackbase serve`: the same required-admin-key
fail-fast behavior, and graceful shutdown on `SIGTERM`/`SIGINT` (stop the listener, close the database,
exit 0).

Immediately after the listener is up, the binary writes exactly one JSON line to stdout:

```json
{"ready":true,"port":3000,"url":"http://0.0.0.0:3000"}
```

This exists for a parent process, like Electron, Tauri, or a supervisor script, to read stdout and know
precisely when the server is ready and which port it actually bound (relevant when `--port 0` or
`$PORT` picks an ephemeral port). Nothing else is printed to stdout before this line. Treat "first line
of stdout" as the contract, not "first line matching some pattern."

</Step>

</Steps>

<Callout type="warn" title="No live hot-swap in a compiled binary">

Unlike `stackbase serve --allow-deploy`, a compiled binary doesn't expose `POST /_admin/deploy`.
Everything embedded at compile time is fixed for the life of that binary:

- Functions and schema: shipping a change means rebuilding and redeploying the binary, not
  `stackbase deploy`.
- The composed component set: adding `@stackbase/workflow` to `stackbase.config.ts` requires a rebuild,
  same as it requires a restart under `serve`.

If you need live hot-swap, run `stackbase serve --allow-deploy` instead (see above). The single binary
trades that away for a zero-dependency, single-file deployment artifact.

</Callout>

### Other limitations

- Single instance. SQLite requires exclusive file access, so it's one binary process per data directory
  (unless you pass `--database-url` to point it at Postgres, in which case the usual single-writer
  advisory-lock rules from [Postgres](/docs/deploy/postgres) apply instead).
- No `stackbase init` scaffolder. You bring your own `stackbase/` directory. `stackbase build` doesn't
  generate one.

### Worked example: fixture app with a component, then cross-compiled

This mirrors the shipped end-to-end test:

```bash title="terminal"
stackbase build --dir stackbase --outfile ./stackbase-server --no-dashboard
# ✓ built ./stackbase-server (58MB)

STACKBASE_ADMIN_KEY=e2e ./stackbase-server --port 3599 --hostname 127.0.0.1 --data-dir ./data
# {"ready":true,"port":3599,"url":"http://127.0.0.1:3599"}

curl -X POST http://127.0.0.1:3599/api/run \
  -H 'content-type: application/json' \
  -d '{"path":"notes:add","args":{"box":"a","text":"compiled"}}'
# {"value":null,"committed":true,...}

# Cross-compile the same app for a Linux server:
stackbase build --dir stackbase --target linux-x64 --outfile ./dist/server-linux --no-dashboard
# ✓ built ./dist/server-linux (95MB)
```

### Going deeper: how the build works

<Accordions type="single">

<Accordion title="How the build works internally">

1. **Refreshes codegen**: `loadConvexDir → push → writeGenerated`, same as `deploy`, so the app's own
   `import "./_generated/server"` resolves once bundled.
2. **Generates a static-import entrypoint.** `bun build --compile` only bundles code reached via static
   `import` statements. There's no filesystem to dynamically scan inside a compiled binary, so
   `stackbase build` writes a throwaway `entry.ts` that statically imports every `stackbase/` module by
   absolute path, the schema, `stackbase.config.ts` (if present), and, if the dashboard is included,
   every dashboard asset via `import x with { type: "file" }`. It then reconstructs the exact
   `{schema, modules}` shape `loadConvexDir` returns at runtime and hands it to `runBinaryServer` (the
   same boot core `stackbase serve` uses, just fed statically imported modules instead of a dynamic
   directory scan).
3. **Shells out to `bun build --compile --minify`** (via `node:child_process.spawnSync`, so this works
   whether the CLI itself is invoked under Bun or Node; only the compile step itself needs Bun on
   `PATH`).
4. **Cleans up the generated entrypoint.**

**Why no `--bytecode`:** the generated entrypoint boots with a top-level `await` (to fully start the
runtime before serving), and `bun build --compile --bytecode` rejects top-level await. Cold-start speed
is negligible for a long-running self-hosted server binary, so `stackbase build` omits `--bytecode`
rather than restructure the entrypoint around it.

</Accordion>

</Accordions>

### Deployment patterns for the binary

<Tabs items={['VPS or bare metal', 'Immutable Docker image', 'Desktop sidecar', 'systemd service']}>

<Tab value="VPS or bare metal">

```bash
STACKBASE_ADMIN_KEY=your-strong-secret ./stackbase-server --hostname 0.0.0.0 --port 8080 --data-dir /var/lib/stackbase
```

<Tab value="Immutable Docker image">

An alternative to the bind-mounted image in [Self-hosting](/docs/deploy/self-hosting): cross-compile
for Linux, then copy just the binary into a distroless base with no Bun, no `node_modules`, and no
source at all.

```dockerfile title="Dockerfile.binary"
FROM gcr.io/distroless/base-debian12
COPY dist/stackbase-server /stackbase-server
EXPOSE 3000
VOLUME /data
ENTRYPOINT ["/stackbase-server", "--hostname", "0.0.0.0", "--data-dir", "/data"]
```

```bash
stackbase build --target linux-x64 --outfile ./dist/stackbase-server
docker build -f Dockerfile.binary -t my-app-binary .
docker run -p 3000:3000 -e STACKBASE_ADMIN_KEY=your-strong-secret -v stackbase-data:/data my-app-binary
```

</Tab>

<Tab value="Desktop sidecar">

Tauri or Electrobun can spawn the binary as a child process and parse its `{"ready":...}` stdout line
to learn the URL to connect to. That gives you process isolation from the app's own runtime.

</Tab>

```ini
[Install]

```

</Tab>

## Choosing between `deploy`, `build`, and Docker self-hosting

| | `stackbase deploy` | `stackbase build` | Docker (bind-mount) |
|---|---|---|---|
| Restart needed to ship a change? | No, live hot-swap | Yes, rebuild and redeploy the binary | Yes, restart the container |
| Schema changes | Additive only, validated, atomic | Whatever the new build embeds (no live validation, it's a fresh process) | Same as `deploy` if you also run `deploy` against it; a plain restart has no gate |
| Component set changes | Requires a restart | Requires a rebuild | Requires a restart |
| Deployment artifact | None, pushes to an existing process | One native executable | Docker image plus bind-mounted `stackbase/` |
| Runtime dependency | `serve` already running | Bun/Node only to build; the binary itself needs nothing | Docker |
| Postgres | Whatever the target `serve` was started with | `--database-url` flag, same as `serve` | Same as `serve` |

In practice these aren't mutually exclusive: you can run `stackbase serve --allow-deploy` in a
container and use `stackbase deploy` against it for day-to-day shipping, falling back to rebuilding the
image only when you need to change the composed component set.
