# Shards B2b Implementation Plan — Multi-Node Shard Distribution

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Shards distribute across nodes — two+ writers commit concurrently on disjoint shard sets via a deterministic rendezvous balancer, per-shard failover/relinquish replaces whole-node exit, and every write (including scheduler/driver-initiated) reaches its shard's owner through one executor-level chokepoint.

**Architecture:** Spec = `docs/superpowers/specs/2025-10-01-shards-b2b-design.md` (post-review, b4e2601 — the review's findings are REQUIREMENTS: the fleet_nodes presence table, the /_fleet/run single-hop guard, writeRouter on ExecutorDeps + convexToJson + boot-step localOnly, releaseShardLock + heartbeatAll RETURNING + the relinquish dispatcher, the armWriter role-arm split + generalized promotion trigger, driver-only stop/start, orphan bump as net-new writer-beat query, point-in-time release). Protocol: verdict.md §b/§d.

**Tech Stack:** TypeScript; rendezvous hashing; PGlite units; Docker-gated multi-node E2E.
