# Shards B2a Implementation Plan — The Multi-Shard Engine (One Process)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Per-shard parallel OCC writers inside one runtime — the `mutation({shardBy})` API, always-on ownership guards at every tier, a per-shard commit-connection pool (true concurrent Postgres transactions), N `shard_leases` rows with per-shard fencing, F = min over frontiers — a single fleet node commits in parallel across shards and every shard mistake errors on the laptop.

**Architecture:** Spec = `docs/superpowers/specs/2025-08-28-shards-b2a-design.md` (post-review, 6ff55bf — the review's findings are REQUIREMENTS: the commit-connection pool with hazards (a)-(c), all-N frontier seeding before ready, the pinned-index scan rule, the two-read-set split, per-slot advisory locks, TableMeta shardKey threading). Protocol: `docs/dev/research/write-sharding/verdict.md` §b. B2b (fleet distribution across nodes) is a separate later slice.

**Tech Stack:** TypeScript; pg per-shard connections; jump consistent hash; PGlite/SQLite units; Docker-gated E2E.
