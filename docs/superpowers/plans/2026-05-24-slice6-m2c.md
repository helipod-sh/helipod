# Slice-6 M2c — Global reactivity (poll-first) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `.global()`/D1-table subscriptions live-update: each global write bumps a per-table version counter in D1; an alarm-driven poller on each DO reads those counters and, when one advances, invalidates that table's subscriptions through the existing `handler.notifyWrites` → re-run → `QueryUpdated` path. Proven cross-DO on miniflare (real DOs + real D1).

**Architecture:** Write side — the M2b global commit flush additionally UPSERTs a `_global_versions` row (atomic with the write). Read side — the kernel records which `.global()` tables a query read (`KernelContext.globalReads` → `UdfResult.globalTables` → `Subscription.globalTables` → a dedicated `byGlobalTable` index). Poll side — an alarm-driven poller reads `_global_versions` for subscribed global tables and calls `runtime.handler.notifyWrites({tables:[T], ranges:[], commitTs})` for each advanced table. The MVCC transaction core, the local reactive path, and the client wire are untouched.

**Tech Stack:** TypeScript; `@stackbase/docstore-d1`, `@stackbase/executor`, `@stackbase/sync`, `@stackbase/runtime-embedded`, `@stackbase/runtime-cloudflare`; miniflare (multi-DO + D1) for the gate.

## Global Constraints (bind every task)

- **Grounding** with verbatim current shapes: `.superpowers/sdd/m2c-grounding.md` (§1 kernel, §2 executor `UdfResult`, §3 sync handler, §4 `SubscriptionManager`, §5 docstore-d1, §6 boot wiring, §7 `observeTimestamp`). **Re-confirm every file:line against live code before editing.**
- **The MVCC core is UNTOUCHED** (transactor/commit/oracle). The poller must NOT call `observeTimestamp` — `.global()` reads never consult the local oracle. It calls only `runtime.handler.notifyWrites(...)`; `commitTs` there is a plain monotone counter/wall-clock used only by the resume-registry's own bookkeeping.
- **Global-table matching uses a DEDICATED `byGlobalTable` index** in `SubscriptionManager` (NOT a merge into `sub.tables`) — a sub reading both a local range and a global table must still match a global-table invalidation.
- **The poller is ALARM-driven** (rides the DO wake/alarm seam — hibernation-safe), NOT a free-running `setTimeout`. It polls only while ≥1 global table has a subscriber; zero subscribed → re-arm nothing (DO can hibernate).
- **Additive:** no D1 binding / no `.global()` table → no `_global_versions`, no poller, behavior byte-identical; the full local reactive suite unchanged.
- **Scope:** equality-only global subscriptions become reactive (matches M2b). NOT fan-out (M2d), NOT range/sorted global queries, NOT the bookmark-replica optimization, NOT CDC/notify.
- Run `bun run --filter <pkg> build` + `typecheck` + the package tests on every task; the E2E is serial-lane (`*-e2e.test.ts`).

```ts
export interface GlobalReactivityDeps {
  readVersions(tables: string[]): Promise<Record<string, number>>;
  subscribedGlobalTables(): string[];
  notifyWrites(inv: { tables: string[]; ranges: never[]; commitTs: number }): Promise<void>;
  now?(): number;
}
export class GlobalReactivityPoller {
  private readonly lastSeen = new Map<string, number>();
  private seq = 0;
  constructor(private readonly deps: GlobalReactivityDeps) {}
  async tick(): Promise<void> {
    const tables = this.deps.subscribedGlobalTables();
    if (tables.length === 0) return;
    const versions = await this.deps.readVersions(tables);
    for (const t of tables) {
      const v = versions[t] ?? 0;
      const prev = this.lastSeen.get(t);
      if (prev === undefined) { this.lastSeen.set(t, v); continue; } // baseline, no invalidate on first sight
      if (v > prev) {
        this.lastSeen.set(t, v);
        await this.deps.notifyWrites({ tables: [t], ranges: [], commitTs: (this.deps.now?.() ?? ++this.seq) });
      }
    }
    // forget tables that no longer have subscribers, so a re-subscribe re-baselines
    for (const t of [...this.lastSeen.keys()]) if (!tables.includes(t)) this.lastSeen.delete(t);
  }
}
```
(Confirm the "first-seen = baseline, no invalidate" choice against the design: a sub's initial result is delivered at subscribe; the poller only needs to catch CHANGES after that. A subscribe that happens between a write and the poller's baseline read still gets the current data at subscribe time, so no missed update.)
