# Slice-6 M2d ‚Äî Cross-shard `fanOut` reads (v1: concat, HTTP-only) ‚Äî Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** On a fixed-shard-count (`mode: "hash"`) multishard deployment, a `fanOut` query fans out to every shard, and the router concatenates the per-shard results into one response ‚Äî non-reactive, HTTP-only, with per-shard failures reported as data.

**Architecture:** Entirely in `@stackbase/runtime-cloudflare-shard` (the ee Worker router). `resolveShard` gains a `"fanout"` resolution kind (gated on `mode: "hash"`, mutually exclusive with a shard key, rejected on WS and for mutations); `worker.ts` fans `stub.fetch` out to `shardIdList(numShards)` with bounded concurrency + per-shard timeout and concatenates each shard's `{value}` array. No engine/executor/client change.

**Tech Stack:** TypeScript; `@stackbase/runtime-cloudflare-shard`, `@stackbase/id-codec` (`shardIdList`); `@cloudflare/vitest-pool-workers` (miniflare multishard E2E).

## Global Constraints (bind every task)

- Grounding with verbatim current code: `.superpowers/sdd/m2d-grounding.md` (¬ß2 route.ts, ¬ß3 errors.ts, ¬ß4 worker.ts, ¬ß5 shardIdList/canonical, ¬ß7 the E2E). **Re-confirm every file:line against current code before editing.**
- **fanOut requires `mode: "hash"`** (a fixed `numShards`). Mode `"key"` has no enumerable shard set ‚Üí rejected `FANOUT_REQUIRES_FIXED_SHARDS`.
- **v1 concatenates** the per-shard `value` arrays (unordered union), applies the query's `limit` to the concatenation. NO ordered merge, NO `mergeSortedAsyncGenerators` extraction (deferred follow-on).
- **HTTP-only** ‚Äî the interface is `POST /api/run?fanout=1` (or `X-Stackbase-Fanout: true`). No `packages/client` change (client is WS-only).
- **Non-reactive** ‚Äî `fanOut` on the `/api/sync` (WS) path ‚Üí `FANOUT_NOT_SUBSCRIBABLE`. **fanOut + a shard key ‚Üí `FANOUT_WITH_SHARD_KEY`.** A fanned-out `.shardBy` **mutation** ‚Üí rejected (fanOut is a read).
- **Failures-as-data** ‚Äî a shard whose `stub.fetch` fails/times out is recorded in `partial.failedShards`, never thrown; the surviving shards' concat is still returned. Bounded per-shard timeout + bounded fan-out concurrency.
- **Additive** ‚Äî a non-`fanOut` request is byte-identical to today; the existing M1 router behavior (explicit/derived/default shard routing, shard-isolated reactivity) is unchanged.
- Run `bun run --filter @stackbase/runtime-cloudflare-shard test` (Node routing unit lane, `vitest.config.ts`), `... build`, `... typecheck` on every task; the miniflare E2E is the separate `test:workers` script (`vitest.workers.config.ts`).

Add the `fanOut` helper (module-level in worker.ts):
```ts
const FANOUT_CONCURRENCY = 8;
const FANOUT_SHARD_TIMEOUT_MS = 10_000;

async function fanOut(ns: DurableObjectNamespaceLike, shardIds: string[], request: Request): Promise<Response> {
  const value: unknown[] = [];
  const failedShards: Array<{ shardId: string; error: string }> = [];
  // Bounded-concurrency pool over the shard list; each shard runs the SAME request (cloned per shard).
  let cursor = 0;
  async function worker(): Promise<void> {
    for (;;) {
      const i = cursor++;
      if (i >= shardIds.length) return;
      const shardId = shardIds[i]!;
      try {
        const stub = ns.get(ns.idFromName(shardId));
        const res = await withTimeout(stub.fetch(request.clone()), FANOUT_SHARD_TIMEOUT_MS);
        if (!res.ok) { failedShards.push({ shardId, error: `shard responded ${res.status}` }); continue; }
        const body = (await res.json()) as { value?: unknown };
        if (!Array.isArray(body.value)) { failedShards.push({ shardId, error: "shard result is not an array (fanOut requires a query returning a list)" }); continue; }
        value.push(...body.value);
      } catch (e) {
        failedShards.push({ shardId, error: e instanceof Error ? e.message : String(e) });
      }
    }
  }
  await Promise.all(Array.from({ length: Math.min(FANOUT_CONCURRENCY, shardIds.length) }, () => worker()));
  // Apply the query's limit to the concatenation, if the body carried one. (Read it from the request
  // body's args if present; else return the full concat. v1: a simple post-concat slice ‚Äî see note.)
  const limited = applyFanoutLimit(request, value);
  return json(200, failedShards.length ? { value: limited, partial: { failedShards } } : { value: limited });
}

function withTimeout<T>(p: Promise<T>, ms: number): Promise<T> {
  return new Promise((resolve, reject) => {
    const t = setTimeout(() => reject(new Error(`shard fetch timed out after ${ms}ms`)), ms);
    p.then((v) => { clearTimeout(t); resolve(v); }, (e) => { clearTimeout(t); reject(e); });
  });
}
```
**On `applyFanoutLimit`:** the router doesn't parse the query's `limit` (it's inside the function body, not the `/api/run` envelope). For v1, the honest simplest correct behavior is to NOT slice at the router (each shard already applied its own `.take(n)` if the query used one, so each shard returns ‚â§n; the concat is ‚â§N¬∑n). Document that v1 returns the full per-shard-limited concatenation and the caller slices if it needs a global cap ‚Äî OR, if the E2E needs a global cap, thread a `?fanoutLimit=` param. **Decide in the implementation:** the simplest correct v1 is `applyFanoutLimit = (_req, value) => value` (no router-side global limit; per-shard limits already applied). Remove the helper and the "apply limit" line if you take that route, and note it in the report + the docs.
