# Critique — DX & Convex-parity lens

Adversarial critic for the client-sync workflow. Lens: **the API surface measured against Convex's actual `withOptimisticUpdate` contract (the migration on-ramp), React ergonomics (hook shapes, re-render behavior, e4's flicker catalog), and the concrete code an app developer writes for `examples/chat`** under each position. Goal is falsification, not improvement. Code claims are file:line against branch `scheduler-component` (client/protocol/handler/chat example re-read 2025-10-16); web/library claims cite e1–e4, which carry the primary URLs. Verdicts marked CONFIRMED (checked against source in this repo or an evidence file's verbatim quote) or PLAUSIBLE (reasoned, residual uncertainty flagged).

Summary of the sharpest results up front:

1. **All three positions ship half of "parity": runtime shape without type inference.** The generated `FunctionReference` has `Returns = any` unconditionally — `localStore.getQuery` cannot be typed — and no position budgets the codegen work. (§1, CONFIRMED)
2. **A and C's own example code violates their own replay contract** (`crypto.randomUUID()`/`Date.now()` inside a function they elsewhere require to be deterministic), producing a real React remount/reorder bug the chat example would exhibit. B is the only position that fixes it — and the fix doesn't require B's registry. (§2, CONFIRMED)
3. **A and C made opposite choices on when the mutation promise resolves, and each claims Convex parity. At most one is parity.** C's divergence is unflagged in C's own text. (§3, CONFIRMED against e1 §2.3's reading of convex-js)
4. **B's two central "closures can't" claims are falsified by C's own design**, and B's registry-as-canonical-surface imposes real on-ramp cost for zero v1 capability gain — B's §1.1 concedes the loop is identical. (§5, CONFIRMED)

## 1. The shared blind spot: type-inference parity is missing from every position, and it's the load-bearing half

Convex's actual DX for this feature is not "a function receives a store." It is: `localStore.getQuery(api.messages.list, { channel })` **returns `Doc<"messages">[] | undefined`, inferred**, and `setQuery` rejects a wrongly-shaped value at compile time — `FunctionReference` carries both args and return types and the store's generics consume them (e1 §2.1; A's own interface sketch at position-a §2 shows `FunctionReturnType<Q>`).

Our generated api cannot instantiate that sketch today:

- `examples/chat/convex/_generated/api.d.ts:3-13` — the generated `FunctionReference<Type, Vis, Args, Returns>` exists, but **every function in the file has `Returns = any`** (`api.d.ts:17-23`), and `Args` is typed only where the argument-validation slice's validators were declared (`messages.send` has typed args at `api.d.ts:22`; `messages.list` is `any, any`).
- The runtime `FunctionReference` in the client is a different, path-only type (`packages/client/src/api.ts:7-9`), and hooks are hand-generic (`useQuery<T = Value>`, `packages/client/src/react.tsx:29`) — the chat example doesn't even use the generated `Api` type; it hand-casts `anyApi` and hand-declares `type Message` (`examples/chat/web/main.tsx:6-8,13`).

Consequence: under **all three positions as written**, the body of an optimistic update is `Value`-soup. A Convex migrant pasting their update function will watch their types silently degrade to `any` — which is worse than an error, because it *looks* like parity until the first refactor. Position C's published interface is the most honest about this, in the worst way: its `OptimisticLocalStore` sketch is literally untyped (`getQuery(ref, args?): Value | undefined`, position-c §3) — the position whose thesis is "parity is the actual product bar" (position-c §1.4) printed the non-parity signature. A sketches the generics but never notes the generated api can't feed them. B types `store` by example only.

None of the three scopes the prerequisite: query **return-type** codegen (explicit `returns` validators, or inference from handler return types threaded through `api.d.ts`), plus re-typing the client's hooks/store generics against the generated `FunctionReference`. That is not part of anyone's "~250–350 LOC client module" (position-a §6) and it is not optional: CLAUDE.md calls the typed-client DX "load-bearing, not a nicety," and this feature is the single API surface where a migrant's existing typed code is pasted in verbatim. **Every position's slice estimate is missing a codegen work item.** (CONFIRMED)

## 2. The determinism contradiction A and C both ship in their own example code

A §4.1 (corollary): update functions "must be cheap, pure, and deterministic over whatever state they find — they run on every ingest." C §3 imposes purity and §5 replays on every ingest. **Both positions' §2/§3 example code puts `crypto.randomUUID()` and `Date.now()` inside the update function** (position-a §2, position-c §3). These cannot both stand. On every applied `Transition`, the rebuild-replay pass re-executes the closure (position-a §3 step d; position-c §5 step 2d) and mints a **new** temp `_id` and a **new** `_creationTime`.

Concrete failure in the chat example: the message list is keyed on `m._id` (`examples/chat/web/main.tsx:39`). While my send is pending, *any* Transition touching `messages:list` — e.g. someone else's message arriving — re-runs my update and gives my pending row a fresh `_id`. React sees a new key → **the pending row unmounts and remounts on every ingest during the pending window** (lost CSS animation state; for a row containing an input or focus, lost focus), and any list sorted client-side by `_creationTime` can reorder the pending row per ingest. This is not a hypothetical class: Replicache documents the exact rule — "unique IDs should often be passed into mutators as parameters, and not generated inside the mutator," because inside-generation yields "a *different* id on every rebase replay" (e3 §1.3). The pattern comes from Convex's own docs (e3 §7: "Temp ids are `crypto.randomUUID()` strings the developer plants"), so "verbatim parity" **imports a documented-example footgun** — parity with the wart included, unmentioned by A and C alike.

B is the only position that fixes it: `placeholderId(mutationId, table, ordinal)` is deterministic across replays and `store.now()` is fixed at entry creation (position-b §1.2). Two things follow, one against A/C and one against B:

- Against A/C: the omission is a genuine falsification of "all 10 e4 failure modes answered." This is flicker-catalog material (a cousin of e4 #2/#4 — identity instability during the pending window) that neither position's catalog pass caught, because the catalog was checked against the *algorithm* and not against the *example code the docs will ship*.
- Against B: the fix does **not** require the registry. A `placeholderId`/`now()` helper pair on `OptimisticLocalStore` needs only the `mutationId` (already assigned at initiation in all three designs) and a per-entry creation timestamp — both available under call-site closures. B presents the helpers as fruits of the log model; they are actually independent, stealable API details. (CONFIRMED)

Related, smaller, and shared: rebuild-replay constructs fresh arrays each pass (`[...existing, {...}]`), so change-detection by reference inequality (all three, following convex-js per e1 §2.2) fires listeners for every update-patched query on every ingest even when the composed value is deep-equal — `useQuery`'s `setState` (react.tsx:36) then re-renders those components. During any pending window, a chat member typing into a busy channel re-renders all patched-query subscribers per incoming message regardless of visual change. convex-js plausibly shares this behavior (unverified); no position mentions it or the cheap mitigation (structural sharing / deep-equal short-circuit on the replayed slice). (PLAUSIBLE, applies equally to A, B, C)

Also shared and unclaimed by anyone: the args-change `undefined` flash in `useQuery` (`react.tsx:35` resets state on key change; flagged by e1 §1.3 as "relevant to optimistic pagination UX"). Switching channels in chat blanks the list for a round trip. Convex has the same property, so it is out of parity scope — but it is the flicker a real chat user notices most, and a slice named "optimistic updates" will collect the bug reports for it. Worth one deliberate sentence of scope-exclusion in whichever position wins; none has it.

## 3. The promise-resolution fork: A and C chose opposite semantics and both call it parity

e1 §2.3, from convex-js `request_manager.ts`, describes `MutationResponse` success **generally** (not only for optimistic mutations): the request "moves to `Completed` carrying the server's commit ts — the optimistic update is not dropped yet, **and the mutation's promise does not resolve yet**." Resolution happens at the ts-gate. So in convex-js, `await mutate(); readLocalState()` is read-your-own-writes for **every** mutation.

- **A** resolves at drop-time (position-a §3 event 4) and flags the behavior change vs today — good. But A's fast-path ("or the update modified no subscribed queries … drop now, rebuild, resolve") plus the silence about mutations *without* an updater means RYOW-await holds only for optimistic mutations that touched subscribed queries. A migrant's `await createThing(); navigate(...)` pattern — no updater attached — behaves differently than on Convex. A claims "Convex parity" for the timing while specifying a conditional version of it. (PLAUSIBLE — e1 doesn't explicitly state convex-js's handling of updater-less mutations, but its request-manager sentence is unconditional.)
- **C** resolves the promise **at `MutationResponse`** (position-c §5 step 3: "markCompleted(requestId, ts); **resolve the promise**"). That preserves today's Stackbase timing (zero behavior change for existing apps — defensible!) but silently diverges from the convex-js contract C's §1.4 holds up as the bar, and C never flags it — while A flags its own opposite choice prominently. An unflagged semantic divergence in the position whose §13 is titled "Uncertainties, flagged" is a process failure, not just a wording one. (CONFIRMED against the two texts + e1 §2.3)
- **B** never says when the success promise resolves at all (position-b §2, H5 success sets `state`/`commitTs` and stops). For a document proposing the canonical public API, the most-called method's completion semantics are unspecified. (CONFIRMED omission)

This matters to the lens because `await`-then-act is the single most common mutation idiom in application code. Whichever position ships must state one rule, uniformly, and its migration note ("same as Convex" / "same as Stackbase-today, differs from Convex in X") — currently no position does this correctly.

## 4. Position A — the cleanest on-ramp, under-audited at its own flagship claim

What survives attack: A's runtime surface is the most faithful to Convex — `useMutation(...).withOptimisticUpdate(fn)`, the three-method store, `getQuery`-returns-`undefined` semantics, `getAllQueries` for paginated arg-families, opt-in per call site, hooks untouched for non-adopters. For the chat example, A is a single ~14-line block at the call site and nothing else changes — the lowest ceremony of the three, and codemod-clean for `stackbase migrate`. The store-level (not component-level) placement is the right answer to e4 #10 and A argues it well (§8, last bullet).

Falsifications and cracks:

- **The §2 example vs §4.1 determinism contradiction** (§2 above). A's no-flicker/no-duplicate proofs (§4.3) survive — the drop is still atomic — but the *pending-window identity churn* is a real, user-visible React defect in A's own headline example, and A's e4 sweep missed it.
- **The conditional-RYOW parity wrinkle** (§3 above).
- **"Hooks structurally unchanged" oversells.** `useMutation` today returns a bare `useCallback` function (`react.tsx:46-52`); attaching `.withOptimisticUpdate` means returning an augmented callable whose identity changes per render when the updater is an inline closure (it always is). Same property as Convex, and harmless in `onClick` — but it is a new re-render/deps footgun for anyone passing the mutation fn into `useEffect` deps or memoized children, and "structurally unchanged" forecloses documenting it.
- **A cites the wrong line for its no-key-mismatch guarantee.** Position-a §2 claims `react.tsx:25` and `client.ts:63` "hash args identically" — but `react.tsx:24-25` (`argsKey`, `|`-separator) is only a `useEffect` dependency key; the store's actual identity is built in `client.subscribe` (`client.ts:63`, `:`-separator). No bug results (the hook delegates to `subscribe`), but the stated proof is wrong as written — the kind of error that matters in a document whose authority is "file:line-verified."
- **A meets, and only meets, e4 #9's mainstream bar.** No pending/failed observability at all ("achievable in userland," §5). True — but B ships `usePendingMutations`/`onMutationFailed` and C ships a pending-count accessor for near-zero cost, and Zero's `.client`/`.server` two-promise DX (e3 §2.2) is evidence users want the affordance. Settling for the bar is a choice, not a law; A frames it as a law.

## 5. Position B — right about two details, wrong about the surface, and its core dichotomy is false

What survives attack: the deterministic `placeholderId`/`now()` helpers (§2 above — the best single DX detail in the corpus), the observable-queue affordances that genuinely exceed e4 #9's bar, and the honest concessions section. B's catalog answers on stacking/rollback are as sound as A/C's — because, as B itself concedes (§1.1, citing e3 §5), the loop is identical.

Falsifications:

- **"Closures cannot express unsent-vs-inflight" — false.** B §2 (transport close): the retention of never-sent entries is "an asymmetry which a closure model can't even express, because it has no entry states." C's `PendingMutation` carries `status: unsent | inflight | completed` *alongside* a non-serialized `update?` closure (position-c §4, S1) and does exactly this retention-and-flush (position-c §7). Entry state is a property of keeping a record per mutation — which A's design also has (`pendingUpdates` with `status`, position-a §3) — not of serializability. B conflates "survives a socket drop" (any in-memory record does) with "survives a process restart" (needs serialization). The claim as written is rhetorically load-bearing for B §9's "layers are a dead end" and it is wrong. (CONFIRMED)
- **"Choosing closures now means rebuilding the public API when offline arrives" — overstated, and B's own sugar proves it.** (a) The wire needs nothing: `requestId` already carries an arbitrary client string (`protocol.ts:46`, verified) under every position. (b) The client's internal record can carry the serializable triple from day one *without* surfacing a registry (C does precisely this), and reload-replay later binds statically-registered updaters by `udfPath` — the registry arriving **when the feature that needs it arrives** (position-c §10 table). (c) Most damning: B keeps `.withOptimisticUpdate` as compat sugar whose entries are "non-durable... excluded from any persisted outbox" (position-b §1.2). So the Convex migrants who arrive via the codemod — the exact population the on-ramp decision optimizes for — get **none** of B's durability benefits until they rewrite to the registry anyway. The rewrite B claims A forces on everyone, B still imposes on its own most numerous users. The day-one registry only pays off for developers who adopt the foreign surface immediately. (CONFIRMED from B's own text)
- **The registry is real ceremony for zero v1 capability.** For the chat example, A/C is one block at the call site; B is: a `defineLocalMutators` module, a changed client construction (`new StackbaseClient(transport, { localMutators })` — `examples/chat/web/main.tsx:11` today takes only a transport), an import of `api` at construction time, and two novel concepts absent from Convex. The effect lives away from the call site that knows the UX context. And per-call-site *control* is lost: under registry-only, every call site of `messages:send` is optimistic, always — Convex lets the compose box be optimistic while an admin bulk-import tool is not. B offers no opt-out mechanism; its #10 "uniformity by construction" is equally "inflexibility by construction." (CONFIRMED as written; B could add per-call opt-out, but hasn't)
- **B's G4 answer trades a 6-line server change for a user-visible staleness class plus a config knob.** A wrong optimistic guess (effect touched Q; mutation didn't invalidate Q) is held "until the touched query's next invalidation" (position-b §2, flagged by B itself) — in a quiet app, a wrong count badge or ghost row **sticks indefinitely on screen**. The escape hatch `dropOnResponse: true` requires the developer to understand ts-gate internals to know when they need it — a knob whose correct use demands exactly the knowledge the abstraction exists to hide. A/C's empty-transition fix bounds the same failure at one fan-out with zero developer knowledge. Under a DX lens this is B's worst call. (CONFIRMED)
- **`pending: true` doesn't typecheck.** B's example plants `pending: true` on a `messages` doc (position-b §1.2) and §8.2 elevates it to the recommended convention — but the value must conform to the query's return type, and `Doc<"messages">` has no `pending` field. Under the typed store all three positions need (§1), the documented convention is a compile error; under today's untyped store it works and silently blocks §1's fix. The pending-row styling story needs a real design (a store-provided wrapper/metadata channel, or a sanctioned type widening); no position has one, but only B builds a doc convention on the gap. (CONFIRMED)
- Unspecified success-promise timing (§3 above).

## 6. Position C — A's surface plus the best cheap wins, undermined by its own sketch and an unflagged divergence

What survives attack: C's developer-facing surface **is** A's (same `withOptimisticUpdate`, same store methods, same call-site code for chat) — so everything good about A's on-ramp transfers. On top, C adds the three best DX-per-line items in the corpus: (1) dev-mode `Object.freeze` on `getQuery` results, converting Convex's documented "will corrupt the client's internal state" footgun (e1 §2.1) from silent corruption into an immediate throw — a genuine beyond-parity improvement for ~5 lines; (2) reconnect + unsent-flush, which is the difference between "message typed during a wifi blip sends" and "error toast" — a *user*-visible UX upgrade, not just architecture; (3) the typed `MutationUndeliveredError`, surfacing SWR's error-dependent-rollback insight (e4 §4) as a type. C also names the seams that make B's future arrive additively, which §5 above shows defuses B's strongest argument.

Falsifications and cracks:

- **The untyped interface sketch** (§1 above) — self-undermining for the "parity is the bar" position.
- **The unflagged promise-timing divergence from convex-js** (§3 above) — the sharpest single hit on C, because C both claims the Convex contract as its bar *and* maintains an "Uncertainties, flagged" discipline it violates here.
- **The inherited determinism contradiction** (§2 above) — C's example is A's example; same remount bug, same silence.
- **Reconnect creates a new, hard-to-document middle state.** Under C's S4 policy, whether your mutation auto-retries after a blip depends on whether it had left the socket buffer — invisible to the app. "Some of your mutations auto-resend, some reject, and you can't predict which" is a strictly harder docs page than A's uniform "disconnect = reject" or a full-dedup "always resends." The typed error makes it *handleable*, not *predictable*. This is a real cost of shipping the "nearly-free" win and C doesn't price it.
- **Scope risk on the one-slice claim.** C's own §13 concedes the ~150-line reconnect estimate rests on an unverified queryId-reuse-across-fresh-session spike, and C uniquely adds G1 hardening to the slice. The honest reading: C is A plus roughly half of another slice. That's a defensible trade — but "two server changes, both protocol-completions" undersells the total surface relative to A's genuinely minimal slice.
- Minor unflagged behavior change: today's `"connection closed"` rejection (`client.ts:237`) becomes `MutationUndeliveredError` for inflight mutations — any existing app matching on the message string breaks. Trivial, but A flagged its equivalent; C didn't.

## 7. What the chat developer actually writes (the concrete comparison)

Today's send path: `main.tsx:17,25` — `const send = useMutation(api.messages.send); void send({...})`, list keyed on `_id` (`main.tsx:39`).

**A / C (identical developer surface):** one block at the call site; render code unchanged.

```ts
const send = useMutation(api.messages.send).withOptimisticUpdate((store, args) => {
  const list = store.getQuery(api.messages.list, { conversationId: args.conversationId });
  if (list === undefined) return;
  store.setQuery(api.messages.list, { conversationId: args.conversationId }, [
    ...list,
    { _id: /* MUST be deterministic — see §2; both positions' examples get this wrong */,
      _creationTime: /* same */, author: args.author, body: args.body },
  ]);
});
```

**B:** the same body, moved to a `defineLocalMutators` map keyed by `api.messages.send.path` in the client-construction module, plus `new StackbaseClient(transport, { localMutators })` (a change to `main.tsx:11`), plus learning `store.placeholderId`/`store.now()`. Net for chat: ~6 extra lines, one extra file touched, two extra concepts — and the effect is no longer next to the `submit()` handler that knows why it exists. B's payoff (`usePendingMutations` for a "sending…" affordance) is real but orthogonal — A/C could expose the same accessor from their pending records.

Under **all three**, the developer must also learn: don't mutate store values in place (C alone makes this a dev-mode throw), the temp `_id` remounts the row at confirm (all three, inherent), and — undocumented by A and C — the temp id must be replay-stable or the row remounts *before* confirm too (§2).

## 8. Ranking

1. **C** — it ships A's exact migration on-ramp (the surface that codemods cleanly and matches what Convex developers' hands already know) while adding the corpus's best cheap DX wins (dev-freeze, unsent-flush, typed ambiguity error) and the seams that neutralize B's rewrite argument; its failures — the untyped sketch, the unflagged promise-timing divergence, the inherited determinism wart — are all fixable in-place without moving any architecture, which is not true of the rivals' failures.
2. **A** — the most faithful and smallest parity slice, and the right scope argument; ranked below C because it is strictly dominated on developer-visible outcomes (no reconnect means the most common real-world event, a network blip, stays a hard error) and its "verbatim parity" flagship is under-audited in its own text (§2's example contradiction, §3's conditional RYOW, the mis-cited hash proof).
3. **B** — ranked last *for v1 canonical surface*, not for lack of insight: the registry imposes ceremony, colocation loss, and a docs-bifurcated on-ramp at the exact feature migrants touch first, in exchange for durability benefits its own compat-sugar users don't receive and that C's seams obtain additively later; its core "closures can't" dichotomy is falsified by C's design; and its G4 knob is developer-hostile. **Two of B's details should be stolen by the winner regardless: deterministic `placeholderId`/`now()` store helpers, and the observable pending/failed queue accessors.**

Cross-cutting, position-independent action items this critique surfaces: (i) return-type codegen is an unscoped prerequisite for real parity (§1); (ii) the docs example must use deterministic placeholder helpers, not `crypto.randomUUID()` (§2); (iii) one uniform, explicitly-stated promise-resolution rule with a migration note (§3); (iv) a typed answer for pending-row styling (§5, `pending: true`).
