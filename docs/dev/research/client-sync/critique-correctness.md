# Critique — CORRECTNESS lens: the reconciliation algorithms vs. the protocol we actually shipped

Adversarial critic in the client-sync workflow. Mandate: falsify, not improve. Method: every load-bearing claim in positions A/B/C re-verified against the code on `scheduler-component` (re-read 2025-10-16: `packages/client/src/client.ts`, `packages/sync/src/protocol.ts`, `packages/sync/src/handler.ts`, `packages/sync/src/session-controllers.ts`, `packages/runtime-embedded/src/runtime.ts`, `packages/transactor/src/shard-writer.ts`, `packages/cli/src/server.ts`), then adversarial interleavings constructed against the real message flow. Verdicts marked **FALSIFIED** (claim is wrong as written), **HOLE** (claim silent on a reachable failure), **HOLDS** (survived attack). Uncertainty flagged where my analysis is reasoning about scheduling rather than a run test.

Summary up front: the shared core loop — apply-at-initiation, replay-survivors-over-fresh-base on every ingest, ts-gated drop — **is sound against today's single-store, tail-serialized fan-out**. I could not construct a corrupting interleaving against the loop itself. What does not survive scrutiny: A's and C's "~6-line" G4 server fix is unimplementable as sited; A's promise-at-drop-time creates a false-reject of committed mutations; C's v1 reconnect reopens a double-apply the ts-gate cannot see; A's and C's own example code violates their own determinism rule (B gets this right); and all three are silent on two protocol behaviors that turn today's benign degradations into sticky wrong UI: backpressure can drop a `MutationResponse`, and the subscribe-vs-notify race can regress the server base *after* the gate closes.

## 1. What survived (verified common ground — no need to relitigate)

- **One wire field.** `MutationResponse` carries no ts (`protocol.ts:59-60`); the handler destructures `commitTs` and drops it at the send site (`handler.ts:209-210`); produced for local and forwarded commits at `runtime.ts:427`. Every position's premise. HOLDS.
- **Origin-in-fan-out is shipped behavior.** `excludeOriginFromTransition` exists (`handler.ts:62-63,253`) and is never set — `runtime.ts:459` passes only `{autoNotifyOnMutation: false, verifyAdmin}`. The origin observes its own write as a `Transition` with `endVersion.ts = invalidation.commitTs` (`handler.ts:273`). HOLDS.
- **Replay-over-base absorbs commit-order inversion.** The server does NOT serialize a session's inbound messages (`cli/server.ts:293,431` — `void handler.handleMessage(...)`, unawaited), so two in-flight mutations from one client can commit in either order. Interleaving I1 (§4) shows replay-in-initiation-order over the authoritative base still converges under inversion. No position claims FIFO commit; none needs it. HOLDS.
- **One Transition per session per commit** (`handler.ts:259-276` groups all affected subs into a single frame), so "drop + authoritative ingest in one notification" is achievable client-side — the no-flicker atomicity claim's precondition is real. HOLDS.
- **OCC-retry-changes-commitTs is a non-event.** The transactor's internal replay resolves before `runMutation` returns; the client only ever sees the final commit's ts. HOLDS — with the `?? 0n` caveat in §3.4.

## 2. Position-specific findings

### F1 — The G4 empty-transition fix is unimplementable as written by A and C. **FALSIFIED (as specced)**

A (§3.1, "~6 lines") and C (§5/§9, "~6 lines after `handler.ts:257`") both site the fix inside `doNotifyWrites`: *"if `originSessionId` is set and absent from `bySession`, send an empty ts-advancing Transition."*

**`originSessionId` never reaches `doNotifyWrites` on any shipped path.** The only call that passes it is `handleMutation`'s inline `notifyWrites(..., session.sessionId)` (`handler.ts:211-213`) — dead code, because the embedded runtime constructs the handler with `autoNotifyOnMutation: false` (`runtime.ts:459`). The real fan-out is the commit-driven drain: `adapter.subscribe((payload) => { queue.push({tables, ranges, commitTs}); void drain(); ... })` (`runtime.ts:656-658`) → `await handler.notifyWrites(inv)` (`runtime.ts:480`) — **no session identity exists anywhere in that payload**. The fan-out is fed by the transactor's commit publish (`shard-writer.ts:370-377`), which fires for commits from every path (WebSocket, HTTP `/api/run`, scheduler drivers, actions' inner mutations) and knows nothing about sync sessions.

So the fix as written is a branch on a parameter that is always `undefined`. The two repair routes both cost more than claimed:

1. **Thread the origin through the commit.** Session id into `executor.run` → transactor → oplog → `adapter.subscribe` payload → drain. Touches the executor options, the oplog shape, and the fleet forward path (`runtime.ts:409-428`). Not 6 lines; it's a cross-package plumbing change through the seam the engine deliberately keeps session-free.
2. **Handler-local map** (`commitTs → sessionId`, armed in `handleMutation`, consumed in `doNotifyWrites`). This has an **arming race I believe is real**: the commit publish fires *inside* `commitWrite`'s stack (`shard-writer.ts:355-377`), before `executor.run`'s promise unwinds back to `handleMutation` — so `notifyWrites` is chained onto `notifyTail` *before* `handleMutation` resumes. In the exact case the fix exists for (origin unaffected → `bySession` empty), `doNotifyWrites` completes in its first microtask, before the map is armed. The workable variant is inverted: `handleMutation`, after sending the response, enqueues an origin-check *on the notifyTail* ("if `session.version.ts < commitTs`, send the empty transition"), relying on the commit's fan-out already being ahead of it in the tail. That ordering in turn depends on the docstore's publish firing synchronously at commit — true for SQLite's in-process path, **unverified for the async `pg` adapter**, where a late publish would make the origin-check run first and close the gate *before* inclusion — the precise flicker the fix exists to prevent.

The *idea* (server guarantees the origin's ts frontier advances past its own commit) is correct and I endorse it. The *siting and cost estimate* in both A and C are wrong, and the correct siting has an adapter-timing hazard neither mentions. This also weakens A's headline "total server delta: 1 wire field + ~6 lines."

### F2 — A's promise-resolution-at-drop-time falsely rejects committed mutations on disconnect. **HOLE (A); avoided by C; unspecified in B**

A (§3 event 4, claim 7) parks the mutation promise until the ts-gate closes. A (§3 event 5) keeps today's transport-close behavior: "Unchanged rejection of all pendings" (`client.ts:235-241` rejects everything with `"connection closed"`). Interleaving: mutation commits → `MutationResponse{success, ts}` received → layer held awaiting the fan-out Transition → socket dies in that window → the promise — for a mutation the client *knows* succeeded (it holds the success response and the commitTs) — **rejects with "connection closed."** The app shows an error for a committed write. Today's client cannot produce this false negative (resolution happens at response, `client.ts:169-177`); A's timing change manufactures it and its event-5 text doesn't carve out completed-awaiting-gate entries. Fix is easy (resolve completed entries at close, drop their layers) — but it is not in the position, and it is exactly the kind of corner a "years of production verification" pitch should have caught. C resolves at `markCompleted` (§5 step 3) and is immune — at the cost of A's await-then-read RYOW claim, a real trade neither position names as a disagreement. B never says when the promise resolves (§2 sets `state:"completed"` and stops) — underspecified on a user-visible contract.

### F3 — C's v1 reconnect reopens a double-apply the ts-gate cannot detect. **HOLE (C)**

C ships reconnect in the same slice (§7). A reconnect is a brand-new server session with `version = INITIAL_VERSION` (`handler.ts:114-119`; sessions are per-socket, `cli/server.ts:280`), and the resubscribe response *preserves* the session's ts: `end = { querySet: start.querySet + 1, ts: start.ts }` (`handler.ts:198-199`) — i.e. **the baseline Transition after reconnect arrives with `endVersion.ts = 0`**, while its *values* reflect the current store, including any write of mine that committed just before the drop.

Interleaving I2: mutation M commits at ts T; `MutationResponse{ts: T}` arrives; layer held for the gate; socket dies before M's fan-out frame; reconnect → resubscribe → baseline with `ts: 0` whose rows **already contain M's write**. Gate predicate (`commitTs <= endVersion.ts`, C §5 step 2; same for `this.version.ts >= ts`, C §5 step 3) sees `T > 0` → layer survives → **M's update replays on top of a base that already includes M** — the message renders twice / the counter increments twice — until some *later* commit advances the session past T. C's §4 `DeliveryPolicy` specifies `unsent` (survive) and `inflight` (reject) but is **silent on `completed`-awaiting-gate entries across a close**. If they're retained, the above fires; if dropped, C is safe (a one-frame flicker at worst) — but the safe rule is unstated, and C's own §13 spike list doesn't include it. A avoids this by scope (no reconnect; event 5 drops all layers); B defers reconnect to slice 2 where its lmid design genuinely does address it. This is the sharpest instance of a general law the positions under-weight: **the ts-gate is only sound over a feed whose ts is monotone for this client; a fresh session resets the frontier, and any design that carries layers across sessions must re-derive the gate** (a client-side `maxObservedTs` carried across the reconnect merely inverts the failure: baseline `ts:0 < maxObservedTs` would mean *nothing* ever gates until new commits arrive — layers stick instead).

### F4 — A's and C's temp-id example code violates their own determinism requirement. **FALSIFIED (their own examples); B is the only position that constructs this correctly**

All three adopt the convex-js contract that update functions are **re-run on every server ingest** (A §4.1 corollary: "cheap, pure, and deterministic"; C §3: "pure over the store"; e1 §2.2: replay is literal). Both A (§2) and C (§3) then show flagship examples minting the temp id **inside the updater**: `_id: crypto.randomUUID()`, `_creationTime: Date.now()`.

Consequences, mechanical: every unrelated Transition re-runs the updater → a **new UUID and a new timestamp per ingest** → (a) the composed value differs by reference *and by content* on every ingest, so change-detection-by-inequality (A §3 step e, C §S2) fires listeners for that query on every frame the client receives; (b) a React list keyed on `_id` **remounts the pending row on every unrelated ingest** — dropped focus/animation/selection on the very row optimism exists to make feel solid; (c) a time-sorted list can reorder the pending row as `Date.now()` drifts. This is precisely the "temp-id swap races with React keys" case my mandate names — and it races not once at confirm (which A §4.3 correctly proves atomic) but *continuously before* confirm. (Convex's own docs share the footgun-shaped example; inheriting a reference bug verbatim is not parity, it's blind copying.) B is the only position that engineered the requirement it states: `placeholderId(mutationId, table, ordinal)` deterministic across replays, `store.now()` fixed at entry creation (B §1.2, §4.1). On this axis B is strictly more correct than A and C, and the fix for A/C is to adopt B's helpers — which their closure API makes awkward (where does the stable per-mutation seed live in a bare `(localStore, args) => …`? A would need to extend the updater signature or the store, unacknowledged).

### F5 — A's sharded-ts safety argument is internally contradictory. **FALSIFIED (the proof, not today's behavior)**

A §3: gate uses `maxObservedTs`, "so the gate always closes no later than the write becomes visible — **the drop can never precede inclusion**." Those two clauses contradict each other: "no later than visibility" *includes* "before visibility." Construct it: shards X and Y with non-monotone interleaved fan-out on one session's feed (the exact case e1 G6 / `write-sharding/evidence-invariants.md` §4 warns about); my write on Y has commitTs 90; a foreign commit on X at ts 100 fans out first → `maxObservedTs = 100 ≥ 90` → my layer drops → **my write is not in the base yet** → my message vanishes until Y's frame arrives. Drop-preceding-inclusion, QED. Today this is unreachable — commits publish through one store with a tail-serialized drain (`runtime.ts:656-658, 466-484`; `handler.ts:241-245`), so arrival order = ts order — but A sells `maxObservedTs` as the thing that makes the design "survive non-monotonic sharded commitTs" (A claim 3), and it does not; it only guarantees the gate *eventually* closes, not that it closes at inclusion. B's forward-compat argument (per-mutation identity confirmation, lmid-shape, is the primitive that survives multi-node — B §2 note, §9.4) is **validated** by this construction. C's `versionCoversCommit()` seam (§9) confines the blast radius to one predicate but a predicate swap alone cannot fix it — inclusion-detection needs new information on the wire (identity or per-shard frontiers), which C half-acknowledges ("revisit … first," §13) and A wrongly claims to have already solved.

### F6 — B's "no server change for G4" bound is not a bound. **FALSIFIED (the "bounded" claim)**

B §2 (G4 fallback): if the entry's effect touched subscribed queries, "the origin's Transition is coming." That inference is exactly wrong in the wrong-guess case — the updater touching Q says nothing about the *mutation's write ranges* intersecting Q's *read ranges*, and range-precise invalidation (`handler.ts:249`) is what decides. B concedes the edge but calls it "bounded staleness of a wrong guess, not a wedge." **The bound is "the next invalidation of Q," which may never come** (quiet table, or a query whose range simply never overlaps future writes): a completed entry replaying a wrong effect over every future rebase, indefinitely, promise long resolved, invisible to the app. That is a wedge by any operational definition — and it is precisely the case A cites as the *reason* the server fix is required ("a wrong-guess-sticks-forever case reachable," A claim 2). B's per-entry `dropOnResponse` escape hatch requires the developer to know their guess is wrong — the one thing they by definition don't. B's trade (zero server changes) is real, but it is a correctness regression relative to A/C's (mis-sited but repairable) server fix, and B's text sells it as parity.

### F7 — B's slice-1 "offline win" is vacuous as specced. **FALSIFIED (the claim's substance)**

B §2/§7: "never-sent `pending` entries safely survive transport close even in v1 … sent on the next connection." B's slice 1 has **no reconnect** (deferred to slice 2, §7) and the log is **in-memory** (durability also slice 2). A `StackbaseClient` whose transport closed is terminal (`transport.ts:55-60`; no reconnect anywhere) — there is no "next connection" on which retained entries could ever drain; the app constructs a new client and the in-memory log is gone. The retained entries are unreachable state. The only real window is the pre-OPEN queue at startup, which the existing transport already handles (`transport.ts:51,63-66`). The state-machine *shape* is fine; the claimed v1 benefit is null.

## 3. Shared holes — reachable in the shipped protocol, addressed by no position

### 3.1 Backpressure can drop a `MutationResponse` → an immortal phantom layer. **HOLE (all three)**

The per-session backpressure controller is "the ONLY way frames leave a session" and it **drops frames of any type** — queue cap drop-newest, sustained-backpressure queue abandonment (`session-controllers.ts:26-28, 74-90`; wired at `handler.ts:115,142-144`). The protocol's stated degradation ("dropped frame → version gap → client resyncs," `protocol.ts:5-8`) repairs **Transitions only**: a dropped `MutationResponse` has no version bracket and no retransmit. Today's cost: a promise that never settles. Under all three designs the cost becomes: an entry stuck `inflight` forever — never gains a `commitTs`, so **no ts-gate ever applies to it** — whose optimistic effect replays over every future base. The user's screen permanently shows a row the server may or may not have (and if the write *did* commit, the confirmed row arrives in the base and the still-replaying updater double-applies it indefinitely — e.g. the message appears twice, forever, on a healthy connection). Slow-client drops are the *designed* path, not an edge. None of the three positions mentions the case. Minimum viable answers exist (exempt `MutationResponse`/`ActionResponse` from droppability — they're small and rare; or expire `inflight` entries on resync), but this must be decided in the slice, and it is a *server* behavior question, further eroding A's "server delta = 1 field + 6 lines."

### 3.2 The subscribe-vs-notify race regresses the server base — and the ts-gate upgrades it from "stale" to "your own write vanishes." **HOLE (all three; C partially credits it)**

`handleModifyQuerySet` and `doNotifyWrites` are not mutually serialized (only notifies serialize with each other, `handler.ts:241-245`). Interleaving I6, fully constructible from the shipped code: MQS with adds Q1,Q2 → Q1's `execSub` resolves (value as of ts 100) and Q1 registers (`handler.ts:186-187`) → while Q2's `execSub` is awaited, a commit at ts 110 touching Q1 drains: `doNotifyWrites` re-runs Q1, sends `Transition{end:(qs,110), Q1@110}`, advances `session.version` → MQS resumes and sends `Transition{start:(qs,110), end:(qs+1,110), Q1@100}` — **stale value, contiguous brackets, no resync**. The base regresses with no protocol-visible signal. e1's G1 records the resync-flavored variant; the race is broader (any concurrent subscribe).

Why this is now load-bearing: let the ts-110 commit be **my own mutation** with an optimistic layer, landing while my resync is mid-flight (resync is the designed backpressure path, and C's reconnect multiplies its frequency). The notify frame closes my gate and drops my layer; the MQS frame then regresses Q1 to the pre-write value — my message, committed *and* optimistically rendered, **disappears** until the next invalidation. All three positions assume "serverValue is authoritative and per-query monotone"; the shipped handler does not guarantee monotone. A explicitly calls G1 "orthogonal … neither worsened nor fixed" (§3) — **falsified**: the layer machinery converts the race's payoff from bounded staleness into the exact artifact (vanish-own-write) the feature exists to prevent, i.e. the slice *worsens the user-visible severity* of the untouched bug. C is the only position that schedules G1 hardening in-slice (§5.5, §10) — correct call, incompletely argued (C also frames it as orthogonal-but-adjacent rather than gate-interacting).

### 3.3 Response-before-Transition ordering is a scheduling accident, not a protocol guarantee. **WEAK SPOT (all three)**

All three arm the gate from the `MutationResponse` and cite the ordering invariant (C §5.3: "the gate is armed before it can fire"; A §1.1; e1 §1.5 citing the `runtime.ts:455-458` comment). The mechanism: the commit publish fires *inside* the commit (`shard-writer.ts:370-377`), so `doNotifyWrites` is chained onto the notify tail **before** `handleMutation` resumes to send the response; response-first holds only because `doNotifyWrites` needs more microtask hops (its `execSub` re-runs) before its `send` than the mutation handler's continuation does. That is relative-async-depth luck, not construction — and the depths differ by adapter (sync SQLite vs async `pg`). If it ever inverts, the Transition carrying my write arrives while my entry is still `inflight` (no `commitTs`) → not dropped → **one-frame double-apply**, then the response arrives and the drop-now rule fixes it. Not corruption — but the no-flicker guarantee all three market as "structural" is, on this edge, conditional. None of the three requires the client to be robust to inversion or pins the invariant with a test; the fix is cheap (on Transition, also gate-check `inflight` entries whose write is provably included — impossible without identity… which is B's point again; or just pin the server ordering with an explicit test + comment contract).

### 3.4 `commitTs = 0` is a reachable wire value → silent ack-drop. **HOLE (all three)**

`runtime.ts:427`: `commitTs: Number(r.oplog?.commitTs ?? r.commitTs ?? 0n)` — the fallback exists precisely because paths have missed the threading before (the comment documents the forwarded-shard fix). If any future path yields `ts: 0` on the wire, every position's "drop when `maxObservedTs >= ts`" clause fires **at the response** — the drop-on-ack antipattern all three cite PowerSync to warn against — silently, with a revert-then-reapply flicker as the signature. A one-line client guard (`ts > 0` else treat as gate-less/hold-until-touched) or a server assertion belongs in the slice; nobody specifies it.

### 3.5 Smaller shared gaps

- **Updater throws during replay** (not at initiation — C handles initiation, §5.1): state shape changed under it (e.g. `QueryFailed` left a stale base, `client.ts:205-213`). Mid-rebuild throw must not leave the composed store half-built; drop-the-entry vs skip-this-replay is unspecified in all three.
- **`client.query()` one-shot** (`client.ts:90-105`) resolves with the *first* value through the subscribe path — under all three designs that's the **composed** view, so a one-shot "read" can return speculative data that later rolls back. Probably desirable; nobody decides it.
- **`QueryFailed` on the confirming Transition**: gate closes (ts advanced), layer drops, but the base kept the *pre-write* value (`client.ts:205-213`) → committed write invisible until the query recovers. A calls the adjacent case "degraded but consistent" (§3); "your committed, optimistically-shown write vanishes" is degraded in exactly the way users notice. Shared, inherent, worth a docs line at minimum.

## 5. Ranking (correctness lens only)

1. **C** — the same verified core loop as A with the fewest self-inflicted specification errors: resolve-at-response avoids A's false-reject (F2), G1 hardening and `versionCoversCommit` show the only position tracking how the gate *interacts* with the shipped races; its one real hole (F3, reconnect double-apply) is closed by a single stated rule (drop non-`unsent` layers at close) and its G4/temp-id defects (F1, F4) are shared with A.
2. **A** — the tightest scope and a sound core, but three unforced errors in the write-up itself: an unhandled false-reject its own timing change creates (F2), a G4 fix that cannot work where it says it goes (F1), and a sharding safety proof that contradicts itself (F5) — for a position whose entire pitch is "proven, precisely, against our exact protocol," the precision is what's missing.
3. **B** — paradoxically the most *carefully engineered* position at the micro level (only one to satisfy the replay-determinism rule all three state, F4; correctly predicts the identity-confirm future, F5/I8) but the worst *shipped-correctness* trade: its no-server-change G4 answer is an unbounded wrong-guess wedge sold as bounded (F6), its headline v1 offline win is unreachable state (F7), and its promise-timing contract is unspecified — the record shape is right about 2027 and underspecified about the slice in front of us.

Cross-cutting verdict for the synthesis: whichever representation wins, the slice is not shippable as specced by *any* position until it decides §3.1 (undroppable responses or inflight expiry), §3.2 (G1-class serialization or re-run-after-register), the real siting of the G4 frontier guarantee (F1 — likely `handleMutation` enqueueing the origin-check on the notify tail, with an adapter-timing test), and adopts B's deterministic placeholder/now helpers regardless of whether B's log wins. Those four are protocol/server questions; the "pure client-side layer" framing all three lean on (A most) is the part of the corpus that did not survive contact with the shipped code.
