# concave + lunora + Stackbase's Reserved Design — File Storage Research

> Researched 2025-07-31. Sources: `.reference/concave-docs-raw`, `.reference/concave-npm`, `.reference/lunora`, Stackbase's own reserved docs (`docs/enduser/configure/configuration.md`, `docs/enduser/deploy/cloudflare.md`, `docs/enduser/deploy/electron.md`, `docs/enduser/deploy/electrobun.md`). Clean-room: concepts extracted only, no concave/lunora code copied into Stackbase packages.

Slice 4 (file storage) is next up. Because concave is Stackbase's direct ancestor — Stackbase's own reserved docs are, section for section, a rebrand of concave's docs (`@stackbase/*` in place of `@concavejs/*`) — concave's design is the load-bearing input here, more than any other reference project. lunora (a Cloudflare-native reimagining of the same Convex-shaped problem) is the second reference, useful for what a *different* substrate (R2 directly, no DocStore abstraction) implies for access control and the upload flow.

---

## 1. What concave actually provided

Concave's positioning is "100% API compatible with Convex" for storage: `ctx.storage.store()` / `.get()` / `.delete()` / `.getUrl()` are all listed **Full** in its compatibility matrix (`llms-full.txt:3047-3054`), with `ctx.storage` explicitly named alongside `ctx.db`, `ctx.auth`, `ctx.scheduler` as the four Convex-parity context surfaces. The docs point at Convex's real `ctx.storage` API docs (`docs.convex.dev/file-storage`, `StorageReader`) as the reference — concave did not invent a different shape, it reimplemented Convex's.

**The storage-id model (from the shipped `@concavejs/core` package, `.reference/concave-npm/ex/core/dist`, not just docs):**

- Storage objects are tracked as rows in a system table, `_storage`, exactly like Convex real. `system/internal.js` shows `ctx.db.system.query("_storage").collect()` and `v.id("_storage")` as the storageId's actual type — a storage ID is a **document ID into a system table**, not an opaque blob-store key. This is the load-bearing fact the Stackbase reserved docs don't currently make explicit (see §4).
- The system table's rows carry `{ _id, _creationTime, sha256, size, contentType, uploadedAt }` — i.e. blob *metadata* lives in the docstore (transactional, queryable, subscribable) while blob *bytes* live in the separate `BlobStore` (fs/S3/R2).
- A `BlobStoreGateway` (`core/dist/kernel/blob-store-gateway.d.ts`) sits between the raw `BlobStore` adapter and the rest of the kernel: it takes `{ docStore, queryRuntime, storage: BlobStore, idGenerator }` and exposes `store`/`get`/`getPublicUrl`/`getMetadata`/`delete` — i.e. it is the piece that *writes the `_storage` row* on every `store()` and reads it back for `getMetadata()`/`getUrl()`. The bare `BlobStore` adapter interface never sees `_storage`; the gateway is what wires bytes-storage to metadata-storage.
- The full syscall surface (`BlobStoreSyscallOps`, `core/dist/kernel/syscalls/api-profile.d.ts`) is bigger than the four adapter methods suggest: `storageDelete`, **`storageGenerateUploadUrl`**, `storageGetMetadata`, `storageGetUrl`, `storeBlob`, `getBlob`, plus legacy-named duplicates (`store`/`get`/`getUrl`/`delete`/`generateUploadUrl`). So concave *did* implement Convex's `generateUploadUrl()` — a mintable, short-lived upload URL — not just direct `store()`.
- **HTTP surface**: `POST /api/storage/upload` (raw body → `{ storageId }`, direct-to-origin-server upload, blob buffered through the Worker/Bun process) and `GET /api/storage/<storageId>` (streams bytes back with `Content-Type`/`Content-Length` set from blob metadata) are reserved, engine-owned paths (`isReservedApiPath` in `http-handler.js` explicitly carves out `/api/storage` and `/api/storage/*` before a user's own HTTP router can shadow them — the same reserved-path pattern Stackbase's own `httpAction` router already uses for `/api/*` and `/_*`).
- **Dashboard/admin integration**: concave ships `systemListStorage` (paginated, sortable by `uploadedAt`/`size`/`contentType`, with a running `totalSize`), `systemDeleteStorage`, and `systemGetStorageMetadata` as internal system queries/mutations gated by `requireSystemCapability("storage:read"|"storage:write")` — this is the backend for a storage/file-browser panel in whatever admin UI concave shipped, directly analogous to Stackbase's already-shipped dashboard data browser.

**Backends** (docs + npm package-name table agree): `FsBlobStore` (Bun/Node local disk — the default for local dev and self-host), `S3BlobStore` (AWS S3 / any S3-compatible endpoint — explicit `bucket`/`endpoint`/`accessKeyId`/`secretAccessKey` constructor options), `R2BlobStore` (Cloudflare R2, default on Workers). **None of these three adapter packages exist in the actual published `@concavejs/*` npm surface we have locally** (`.reference/concave-npm/ex` lists only `docstore-memory`, `docstore-sqlite-base`, `docstore-node-sqlite`, `runtime-base`, `runtime-embedded`, `cli`, `core`, `brand`, `transport-capnweb` — no `blobstore-*` package). The `BlobStore` interface, the `BlobStoreGateway`, and the syscall plumbing are real, shipped code in `@concavejs/core`; the concrete fs/s3/r2 adapters are documented but **not present in what we have** — i.e. concave got as far as *reserving the seam and wiring the kernel side* but the adapters themselves are either unpublished, in a different package we don't have, or (most likely, matching Stackbase's own current state) planned-but-not-yet-shipped at the point this reference was captured.

**Local dev storage location**: `.concave/local/` for both SQLite and files (per the "Common questions" FAQ) — desktop embeddings (Electron/Electrobun) instead use `{userData}/concave/storage/` and `~/.my-app/storage/` respectively, mirroring the SQLite db path convention 1:1 (`db.sqlite` next to `storage/`).

**Anything concave did differently from Convex real:** nothing surfaced in the docs or shipped `core` package suggests a deliberate divergence — the whole framing is "identical to Convex," and the one open compatibility gap called out elsewhere in the same docs (crons "Planned", Convex Auth "Not supported") does *not* list storage as a gap. The `_storage`-system-table + `BlobStore`-byte-backend split is itself inherited from Convex's real architecture (Convex's own file storage is metadata-in-the-transactional-log, bytes-in-S3), not a concave invention — concave's contribution was making the *byte backend* pluggable (fs/s3/r2) where Convex Cloud hardcodes S3.

## 2. The reserved Stackbase `BlobStore` seam

Stackbase's `docs/enduser/configure/configuration.md` (and the `cloudflare.md`/`electron.md`/`electrobun.md` deploy docs) already document this design — verbatim-rebranded from concave's own docs, package names swapped `@concavejs/*` → `@stackbase/*`:

```ts
// docs/enduser/configure/configuration.md
export interface BlobStore {
  store(blob, options): Promise<{ _id: string; sha256: string; size: number; uploadedAt: number }>;
  get(storageId): Promise<Blob | ArrayBuffer | null>;
  delete(storageId): Promise<void>;
  getUrl(storageId): Promise<string | null>;
}
```

(The concave-npm `core/dist/abstractions/blobstore.d.ts` shows the fuller, typed version of the same shape: `StorageId { id: string }`, `StorageMetadata { _id, sha256, size, contentType?, uploadedAt }`, `StorageOptions { contentType?, storageId? }`.)

**Metadata shape and the sha256 implication.** `{ _id, sha256, size, uploadedAt }` returned from every `store()` call means:

- `sha256` is computed **by the adapter, on every store**, not optionally — every `BlobStore` implementation must hash the bytes it's given. This is the natural hook for **content-addressed dedup**: if two `store()` calls produce the same `sha256`, an adapter (or the gateway layer above it) *could* skip writing a second physical copy and just point a new `_id` at the same underlying bytes. Nothing in the reserved design says whether Stackbase should actually do this — it only guarantees the hash is always available (see gap in §4).
- `_id` is adapter-assigned (an opaque storage identifier), separate from `sha256` — so two different `_id`s can share one `sha256` without a design conflict; `_id` is the "logical upload," `sha256` is the "these bytes are identical" signal.
- `uploadedAt` is a Unix-ms timestamp, set once at store time, never mutated — matches the log/document immutability the rest of Stackbase's docstore already assumes (append-only MVCC), so a `_storage` row (if Stackbase adopts one, see §4) would slot into the existing document model with no special-casing.

**The fs/s3/r2 adapter split.** Config wiring (`stackbase.config.ts` and the programmatic `StackbaseOptions`/`StackbaseCfRuntimeOptions`) already reserves three phantom packages exactly mirroring concave's:

| Package | Target | Status |
|---|---|---|
| `@stackbase/blobstore-bun-fs` | Local disk, Bun/Node self-host — the default | reserved, not built |
| `@stackbase/blobstore-bun-s3` | S3-compatible object storage, production self-host | reserved, not built |
| `@stackbase/blobstore-cf-r2` | Cloudflare R2, Workers | reserved, not built (moot until a Cloudflare runtime tier exists — Stackbase currently has no CF adapter shipped at all, per CLAUDE.md's "Tier 2 distributed sharding/sync fleet" being deferred) |

Given Stackbase's locked decisions (Bun-primary, Docker self-host baseline, Postgres deferred to 6c, no CF tier yet), **only the fs adapter is actually load-bearing for Tier 0** — s3 is the natural "production self-host, don't want files on the container's local disk" upgrade path (same role S3 plays for anyone self-hosting today), and r2 stays aspirational until/unless a Cloudflare tier gets built.

**How this mirrors the shipped `DocStore` seam.** Stackbase already has a narrow, driver-agnostic seam for documents — `packages/docstore` (`export interface DocStore { setupSchema, write, get, index_scan, load_documents, previous_revisions, scan, count, maxTimestamp, getGlobal, writeGlobal, writeGlobalIfAbsent, close }`), with `packages/docstore-sqlite` and `packages/docstore-postgres` as the two shipped adapters, and the engine (`packages/transactor`, `packages/query-engine`) importing only the interface, never a driver. `BlobStore` is the same shape of decision one level up: a `packages/blobstore` interface package + `packages/blobstore-fs` / `packages/blobstore-s3` adapter packages, with nothing in the engine core ever importing `node:fs` or an S3 SDK directly. The CLAUDE.md rule "Never let the engine know which database it's on" extends cleanly to "never let the engine know which disk/bucket it's on."

One structural difference worth flagging: `DocStore` is a **required** engine dependency (every deployment has exactly one docstore) where `BlobStore` is documented as **optional** everywhere in concave's docs ("Do I need R2? Only if your app uses `ctx.storage`") — `blobstore` is nullable in `StackbaseOptions`, and the HTTP storage routes 500 with "Storage not configured" rather than crashing at boot when absent. Stackbase's reserved design already carries this optionality forward correctly.

## 3. lunora's approach — R2 + Workers/Durable Objects

lunora takes a structurally different path: no docstore/blobstore split at all, because lunora's storage substrate for *documents* is already Durable Object SQLite/D1 (bespoke to Cloudflare), and blobs go straight to **R2**, wrapped by a dedicated `@lunora/storage` package (`packages/storage/src/{create-storage,bucket-storage}.ts`) rather than threaded through a generic pluggable-adapter interface. Concretely:

- **`createStorage({ bucket, publicBaseUrl?, signingSecret?, s3? })`** wraps a raw R2 binding into a `Storage` object exposing `store`/`upload` (alias), `download`, `delete`, `getMetadata`, `list`, `getUrl`, `getSignedUrl`, `generateUploadUrl` (Convex-name alias for a signed `PUT`), `getPresignedUrl` (native S3-compatible presigned URL, bypassing the Worker entirely, requiring R2's S3 credentials), and multipart-upload primitives (`createMultipartUpload`/`resumeMultipartUpload`) for very large objects.
- **Read/write split is enforced by function-context type, not by convention**: `download`/`getMetadata`/`getUrl`/`getSignedUrl` are available from `query`, `mutation`, *and* `action`; `generateUploadUrl`, `store`, and `delete` are **action-only**. The stated rationale is that signing is a pure HMAC computation (no R2 round-trip, so safe from a read-only/transactional context) while an actual R2 write/delete is a side effect that can't participate in or be rolled back by a transaction — so it's forced out to actions, the same non-transactional escape hatch Stackbase already built for its own `action` functions. This is a stronger, type-level guarantee than concave/Convex's `ctx.storage`, which is nominally available everywhere but only *actually* side-effect-free in queries by convention.
- **Presigned/signed uploads are the recommended pattern, not an edge case**: the docs walk through an action minting a short-lived signed `PUT` URL (`generateUploadUrl`), the browser uploading directly to R2 (the Worker never proxies the bytes), then a follow-up mutation recording the key in the database — i.e. a deliberate **two-phase upload**: (1) get-a-URL, (2) client-to-bucket direct PUT, (3) record-the-reference. This keeps large file bytes off the origin server/Worker entirely, which the concave/Convex direct-upload-through-origin path (`POST /api/storage/upload`) does not.
- **Keys, not opaque IDs.** lunora addresses objects by a caller-chosen string `key` (`avatars/${userId}/${key}`), not a generated storage ID — closer to raw S3/R2 semantics than Convex's system-table-row model. `scopeKey(prefix, key)` is an explicit helper for per-tenant key namespacing to prevent IDOR, and `validateKey` rejects path-traversal (`..` segments), NUL bytes, and over-length keys — hard lessons from letting user input reach a real key namespace directly (concave/Convex's opaque generated IDs sidestep this whole class of bug by construction).
- **Access control is a first-class DSL**, not left to hand-rolled checks: `defineStorageRule({ bucket, on: "read"|"write"|"delete"|"list", prefix?, when })` composed via `storageRules([...])` middleware — the object-storage analogue of row-level-security rules over documents — statically discoverable by codegen (`discover-storage-rules.ts`) and surfaced read-only in the studio dashboard's access-rules view.
- **Named/multiple buckets** are supported via `createBucketStorage({ default: ..., avatars: ... })`, with `ctx.storage.bucket("avatars")` scoping every operation, typed against the schema so an unknown bucket name is a compile error.
- **Dashboard tooling goes further than a plain file browser**: `packages/studio/src/features/storage/file-orphans.tsx` implements *bidirectional* orphan detection — dangling references (a `v.storage()` schema field pointing at a bucket key that no longer exists) and orphaned objects (bucket keys with no `v.storage()` field referencing them) — something neither concave nor (as far as documented) Convex Cloud's own dashboard does.

The core lesson from lunora, given it targets a hard real substrate (actual R2, actual Workers) rather than a doc-only design: a **presigned/signed two-phase upload** and an **explicit action-only write gate** are what a production file-storage feature needs beyond the bare adapter interface, and **key validation is not optional** the moment any part of the key is caller-influenced.

## 4. Gaps/ambiguities in the reserved design

The reserved `BlobStore` interface in `docs/enduser/configure/configuration.md` is a faithful copy of concave's *adapter* interface — but concave's actual file-storage feature is bigger than that one interface, and Stackbase's reserved docs stop at the adapter boundary. Concretely, still unspecified:

1. **No metadata table (`_storage`) is reserved anywhere in Stackbase's docs.** Concave's real design (confirmed in shipped `@concavejs/core`, not just docs) tracks every blob as a row in a `_storage` system table — `v.id("_storage")` is the actual storageId type, queryable via `ctx.db.system.query("_storage")`. Stackbase's schema/system-table design (`packages/docstore`, `packages/component`, whatever powers `_admin:browseTable`) has no reserved `_storage` table. Without it: no way to list/paginate/query storage metadata transactionally, no way for a mutation to atomically both write an app-row and record which storage ID it references (i.e. no read/write-set tracking → no reactive invalidation → a query watching a blob's existence can't be a normal Stackbase reactive query), and no natural backing for a `systemListStorage`-equivalent dashboard file browser (which the already-shipped Stackbase dashboard data browser, `_admin:browseTable`, would otherwise extend to for free if `_storage` were just another system table).
2. **No `ctx.storage` runtime API is reserved.** The interface today is purely the *adapter* (`BlobStore`), i.e. what a backend implements. Nothing specifies the function-context surface an app author actually calls — is it `ctx.storage.store(blob)` inside a mutation (breaking the "mutations are the only writers, but must be deterministic/no side effects" invariant CLAUDE.md locks in, since a `store()` call to fs/S3/R2 is exactly the kind of non-deterministic side effect actions exist to fence off)? Or, per both concave (Convex-parity: `ctx.storage` fully available, no context restriction beyond "planned/full" table entries) and lunora (action-only write, read-anywhere) — which shape does Stackbase pick? Given Stackbase already ships `action` as the non-deterministic-side-effect escape hatch (see CLAUDE.md's Action runtime section) and already has the query/mutation-must-be-deterministic invariant locked in, **lunora's action-only-write pattern is the one consistent with Stackbase's own existing architecture** — a mutation calling `ctx.storage.store()` directly would violate the same determinism/replay invariant that keeps queries and mutations out of `fetch`/`Date.now()`/`Math.random()`.
3. **No upload-URL HTTP endpoint is reserved.** Concave reserves `/api/storage/upload` (direct-body POST) and a `generateUploadUrl` syscall; lunora's two-phase signed-PUT pattern needs no server proxy at all for the byte transfer. Stackbase's own `httpAction` + public HTTP router (already shipped, per CLAUDE.md) reserves `/api/*` and `/_*` — a storage upload endpoint would need to either reuse that reservation or add a new one, and today's reserved-path guard (`route()` rejecting `/api/*`/`/_*` registrations) has no test or doc coverage for a `/api/storage/*` carve-out the way concave's `isReservedApiPath` explicitly does.
4. **No access control model is specified at all.** `getUrl()` in the reserved interface returns a URL with no notion of expiry, signing, or scoping — concave's own docs call this "signed URLs" in the compatibility table but the reserved `BlobStore.getUrl(storageId): Promise<string | null>` signature has no options bag for `expiresInSeconds` or public-vs-private. lunora's `storageRules` DSL (bucket/operation/prefix/`when`-predicate) is a much stronger answer to "who can read/write/delete which objects" than anything currently reserved for Stackbase — right now, any caller holding a storageId can apparently call `getUrl`/`get`/`delete` with zero authz hook, which is a real gap once storage ships alongside Stackbase's existing `effectivePermissions`/authz slice.
5. **Whether to keep sha256 content-addressing is unresolved.** The metadata shape mandates computing `sha256` on every store, but nothing says whether Stackbase should *act* on it — i.e. whether two `store()` calls with identical bytes should physically dedup (one blob, two `_storage` rows pointing at it) or just carry a redundant hash for integrity/audit purposes only. Concave's shipped code computes and returns the hash but nothing in the accessible source shows dedup logic actually consuming it. This is a real design fork: dedup saves disk/egress on the fs/s3 adapters but adds a refcounting problem on delete (can't delete physical bytes until the last referencing `_storage` row is gone) that the reserved interface's simple `delete(storageId): Promise<void>` doesn't accommodate.
6. **No orphan/dangling-reference tooling is reserved**, unlike lunora's bidirectional check (`file-orphans.tsx`). Given Stackbase already ships a dashboard data browser, this is a natural, cheap extension once `_storage` exists as a system table — but it's presently unplanned.
7. **No key-validation/path-traversal story.** The reserved interface takes an opaque adapter-assigned `storageId`, so lunora's whole `validateKey`/`scopeKey` concern (caller-supplied keys enabling path traversal or cross-tenant IDOR) may be moot *if* Stackbase keeps IDs opaque and adapter-generated — but if the fs adapter is naively expected to lay files out under a caller-influenced path (e.g., mirroring `contentType` or the storageId string itself into a filesystem path), the same class of bug is live. Worth deciding explicitly rather than by omission.

## Implications for Stackbase

The reserved `BlobStore` seam is a real head start, not a false start: it is concave's own actual adapter interface, already rebranded and load-bearing in Stackbase's public docs, and it already reserves the fs/s3/r2 backend split that mirrors the shipped `DocStore` pattern almost exactly (`packages/docstore` + `docstore-sqlite`/`docstore-postgres` → `packages/blobstore` + `blobstore-fs`/`blobstore-s3`, with r2 deferred until a Cloudflare tier exists). Slice 4 should mirror that shipped `DocStore` package structure directly: a `packages/blobstore` interface package the engine core depends on, `packages/blobstore-fs` as the Tier-0 default (parallel to `docstore-sqlite`), `packages/blobstore-s3` as the production self-host upgrade (parallel to `docstore-postgres`), and r2 left reserved-but-unbuilt like the rest of the Cloudflare tier.

What the design still needs, concretely, before slice 4 can be spec'd:

- **A `_storage` system table** — the single biggest gap. Blob *metadata* (`sha256`, `size`, `contentType`, `uploadedAt`) belongs in the transactional docstore as ordinary documents (giving it read/write-set tracking, reactive subscriptions, and dashboard browsability for free via the existing `_admin:browseTable` machinery), exactly as concave/Convex do it. The `BlobStore` adapter should stay dumb (bytes in, bytes out, sha256 computed) — a gateway layer above it (concave's `BlobStoreGateway` is the concept to extract, not copy) should own writing/reading the `_storage` row.
- **A `ctx.storage` API, scoped to actions** — not queries/mutations. This is the one place Stackbase should *diverge* from concave's Convex-literal "ctx.storage everywhere" and instead follow lunora's action-only-write / read-anywhere split, because it's the only shape consistent with Stackbase's own already-locked determinism invariant (queries/mutations must not touch the non-deterministic world; actions already exist as the escape hatch). `getUrl`/`getMetadata` (pure reads plus signing math, no I/O side effect) can stay available from queries/mutations the way lunora allows.
- **An upload-URL HTTP endpoint** — reserve `/api/storage/*` (mirroring concave's) or fold it under the existing `httpAction` router's `/api/*` reservation, with the same registration-time reserved-path guard `httpAction` already enforces. Decide between concave's direct-body-through-origin upload and lunora's two-phase signed-URL pattern (or support both — direct upload for the fs adapter where there's no bucket to redirect to, signed URL for s3).
- **An access-control layer** — at minimum, expiring/signed `getUrl()`, and ideally something in the spirit of lunora's `storageRules` DSL once Stackbase's own `effectivePermissions` authz slice extends to blobs, not just documents.
- **A decision on sha256 content-addressing**: keep it as an integrity/audit-only hash (simplest — delete stays a straight 1:1 operation), or build real dedup with refcounting. Given Stackbase's bias toward shipping a complete, working system over a maximally clever one, the audit-only route is the safer default for a v1 slice 4, with dedup as a documented future optimization rather than a day-one requirement.
