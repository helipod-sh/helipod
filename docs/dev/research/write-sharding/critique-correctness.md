# Adversarial review — lens: CORRECTNESS & FAILURE MODES

Verified against source before writing: `packages/transactor/src/single-writer-transactor.ts` (mutex/ring/snapshot discipline as the audit describes), `packages/docstore/src/timestamp-oracle.ts` (plain in-memory counter — today's `_creationTime` is a **commit counter**, not ms), `packages/executor/src/kernel.ts:172` (`_creationTime: Number(ctx.snapshotTs)`), `ee/packages/fleet/src/replica-tailer.ts` (watermark = `maxTimestamp()` pull, ts-group atomic apply), `ee/packages/fleet/src/lease.ts` (advisory lock is the real mutual exclusion; `fleet_lease` is discovery; **epoch bumps only on `tryAcquire` success**, i.e. only after the old session is gone), `ee/packages/fleet/src/node.ts:426–434` (promotion order, `observeTimestamp(maxTimestamp())`), `packages/docstore-postgres/src/postgres-docstore.ts` (write txn shape, `_shardId` dropped).

The single most important verified fact for what follows: **in the shipped code, "lease expired" and "writer fenced" are different events.** The epoch changes only when a successor wins the advisory lock, and the advisory lock is session-scoped — it outlives lease expiry for as long as the old writer's Postgres session lives. Two of the three designs build their safety argument on conflating these.

---

## Design A — "One Line, Many Hands"

### (a) Strongest attack: the live-lease exclusion rule is unsound — the skipped-ts-99 bug is not "structurally dead," it's relocated to failover

A's safety claim (§1.2): every row with `ts ≤ F` is present and nothing will ever land at `≤ F`, where `F = min(frontier_ts)` **over shards with a LIVE lease**. The proof sketch handles the no-lease case with: *"s cannot commit at all (every commit requires the epoch-matched lease-row update…; the promotion path in §6 guarantees no in-flight zombie transaction survives)."*

That sentence is false in the exact window that matters. Concrete interleaving:

1. Writer W holds shard 3's lease (epoch 7) and advisory lock. It opens a commit transaction: `nextval()` → ts **99**, executes the INSERTs and the epoch-conditional `UPDATE shard_leases … WHERE epoch=7` (epoch is still 7 — the UPDATE matches). All statements complete; the transaction is now **idle-in-transaction**, waiting for the process to send `COMMIT`.
2. W's event loop stalls (GC pause, container CPU throttle, TCP hiccup on the app side). Heartbeats stop. `expires_at` passes. Shard 3's lease is now **not LIVE**.
3. Per A's own rule ("F excludes the dead shard's frontier so fleet visibility keeps advancing"), every reader drops shard 3 from the min. F advances to, say, **150** on the other shards' frontiers. Tailers pull `(wm, 150]`, advance `wm = 150`. `StablePrefixTs` brands 150 as provably stable.
4. W wakes and sends `COMMIT`. Nothing stops it: the advisory lock is still held by its live session, so **no successor has promoted and the epoch is still 7** — the conditional UPDATE already matched. `statement_timeout` does not fire (that kills long *statements*; this txn is idle between statements — the relevant guard would be `idle_in_transaction_session_timeout`, which the design never sets). Ts **99 lands below F = 150**.
5. Every replica's watermark is already at 150. The next pull is `(150, …]`. **Ts 99 is skipped on every replica, forever.** MUST-HOLD #1 violated; a "provably stable" branded frontier served torn state; the per-shard `prev_ts` density check the design proudly adds would detect the corruption *after the fact* and crash the tailer fleet-wide — turning silent corruption into a fleet-wide crash loop, which is better, but is not the claimed "structurally dead."

The root cause is precise: **exclusion from F is triggered by lease expiry (a clock), but commits are fenced by epoch (a lock transfer), and the lock transfer cannot happen while the wedged session lives.** The design's promotion fencing is genuinely sound — Postgres kills in-flight txns when the session dies, and the epoch predicate kills stragglers after a real promotion — but the min-exclusion path bypasses promotion entirely. Note the writer self-exit-on-lease-loss (`LeaseMonitor`) doesn't save this: a paused process can't self-exit during the pause, and the `COMMIT` races the exit.

The fix exists (only exclude a shard from F after a *completed fence* — e.g. any node may run `UPDATE shard_leases SET epoch = epoch+1 WHERE shard_id=s AND expires_at < now()`, which serializes against the in-flight commit's row lock on the same row and forces exactly one of {straggler commits first and its frontier counts, fencer wins and the straggler aborts}). But the fix concedes the thing A markets against Design C: **a wedged writer now stalls F for the full wedge duration** (the fencing UPDATE blocks behind the idle transaction's row lock until Postgres or an operator kills the session), not "drops out of the min entirely." A's headline "staleness degrades latency, never correctness" is only true after adopting the stall it claimed to avoid.

Secondary attacks, briefly:
- **RYOW regression is client-error-shaped, not just latency-shaped.** The scalar wait clears only when *every* shard's frontier passes commitTs. The shipped forwarder waits with a **5000ms timeout** (`forwarder.ts`, confirmed by the audit). One stuck commit txn or lease flap anywhere → F stalls > 5s → RYOW timeouts fleet-wide → clients read below their own acknowledged writes (MUST-HOLD #6 breached in degraded mode) or get errors, for writes on perfectly healthy shards.
- **Two consistency levels for the same table.** A global table read from a default-shard mutation is serializable; the same table read from a shard-3 mutation is an un-validated F-read up to ~H+lag stale. Since identities/auth live on the default shard, every sharded mutation's auth check is now a stale, non-OCC-validated read — a revoked permission stays effective on all non-default shards for the staleness window. The design documents write skew abstractly but never names the auth-staleness instance, which is the one that ships CVEs.

### (b) Honesty of stated weaknesses

Weaknesses 1–3 are real and well-chosen; the write-skew disclosure is genuinely honest. But the design **overstates its central safety claim** — "structurally dead, not patched," a proof sketch with a hole exactly at the failover step, and a §6 that asserts the hole closed ("closed at the store, not by heuristics") when the min-exclusion path never reaches the store's fence. That's the worst kind of understated weakness: not an omitted cost, an incorrect proof.

### (c) Steelman

"The exclusion hole is a one-paragraph amendment, not an architecture flaw: make eviction fencing-first (epoch-bump-then-exclude, serialized by the lease row's own lock — the row is already the fence), set `idle_in_transaction_session_timeout` as belt-and-suspenders, and F's worst-case stall becomes bounded by the same lease-TTL every design pays at failover. Everything else — one line, scalar StateVersion, unchanged client, cross-shard subscriptions as literal prefix reads, the deferred version-gap resync kept alive — survives intact, and no rival offers that combination. B pays for its immunity with a permanently weaker product (effect-before-cause across groups); I pay with a bounded, observable stall."

That steelman is strong. It is also an admission that A's failure-mode marketing versus B was wrong.

## Design B — per-shard logs, frontier vectors

### (a) Strongest attack: the "consistent cut" theorem is false under B's own frozen reads — and the anomaly lands squarely on the headline feature

B's load-bearing claim (§0, §4.2): *"because every transaction reads and writes exactly one shard, any vector of per-group closed frontiers is a consistent cut… no visible write can depend on an invisible one **through the database itself**."*

B's own §2 then adds frozen reads: a group-g mutation may read group 0 (and, opt-in, any group) via `ctx.db`. That **is** a read through the database. The cut theorem holds only for write-write dependencies; the moment a mutation's write depends on a foreign-group read — which B explicitly supports, and which the mandatory case (config, identities, "look up the user while writing the message") guarantees will be ubiquitous — the vector can order effect before cause:

- Admin (session 1) flips `channels.archived = true` on group 0's line at `ts_A`.
- A group-5 mutation frozen-reads it (or doesn't — even simpler: user posts "reply to my message in #general" — the #general message is on group 2 at `ts_B`, the reply on group 5 at `ts_C`, `ts_C` caused by `ts_B`).
- A third client's subscription is served at vector `(…, W_2 < ts_B, …, W_5 ≥ ts_C, …)` because group 2's tail on that node is 800ms behind (NOTIFY dropped, 1s poll). The reply is visible; the message it replies to — committed earlier, on the same screen — is not.

Under A and C this is impossible by construction (one line, prefix reads). B's mitigation — per-session monotonic floors — only protects causality *the reading session itself created*. Cross-session causality (the overwhelming case in a collaborative app, which is Stackbase's demo genre) is "eventual with lag-bounded skew," where the bound is **not bounded**: it's local-replica tail lag per group, which under a tailer stall is arbitrary. B's §4.6 does disclose this — credit — but frames it as an exotic corner while it is in fact a first-class product behavior of the one feature ("live cross-shard subscriptions, the piece no ancestor built") all three designs compete on. B ships that feature with visibly torn causality; A and C ship it with a true snapshot.

Second attack: **frozen-read staleness is unbounded and security-adjacent.** A mutation's group-0 read comes from *the executing writer node's local replica* at `W_0`. That is: mutation semantics now depend on a background tailer's health. Auth/identity tables are unsharded ⇒ every sharded mutation's permission check is as stale as that node's group-0 tail — seconds under a stall, with no ceiling, and *silently* (no error, no metric named in the design). A at least bounds the same anomaly at ~H + commit latency. Related wrinkle B never mentions: if the executing node *also happens to hold group 0's lease*, those reads could be served fresh from the primary — mutation-observable behavior that varies with lease placement, i.e. with which node the balancer put a lease on. Same code, different anomaly surface per topology.

Third (smaller): the per-row frontier filter is waved through as "a `shard_id` filter per scan." It isn't — a scan over a sharded table's index must evaluate `ts ≤ W[shard_of(row)]` **per revision**, with `shard_of` varying row-by-row within one index range. That's a join against a P-row frontier table inside the hottest query path in the system (replica MVCC scans), touching `docstore-sqlite`'s latest-revision-≤-ts queries, the query engine's overlay logic, and pagination — the widest query-engine diff of the three designs, priced at one sentence.

Where B is genuinely strong under this lens, verified: the wedged-writer scenario that breaks A is a non-event here. The frontier *is* the commit's own fenced row-update; there is no min over anything; a stalled idle-in-transaction commit merely stalls **its own group's** `last_commit_ts`, and when it finally commits, `W_g` advances to include it — dense by construction, blast radius exactly one group. The `observeTimestamp(maxTimestamp(g))` promotion seed is protected the same way today's is (session death aborts in-flight txns before the advisory lock is obtainable), with the epoch UPDATE as real belt-and-suspenders. B is the only design with no min-exclusion trap to mis-specify.

### (b) Honesty of stated weaknesses

The most honest of the three overall — weakness 1 says "the rival can honestly claim a stronger story here" out loud. Two miscalibrations: weakness 2 (`_creationTime` migration) is **overstated** — today's `_creationTime` is a monotone commit counter (verified: oracle counts from 0), so old counter values sort strictly below new wall-clock ms values and existing index keys need no rewrite; the migration is far less scary than "botched inside every index." Meanwhile the *understated* items are the unbounded frozen-read staleness (incl. auth), the lease-placement-dependent read freshness, and the query-engine vector plumbing. Net: honest on strategy, thin on read-path mechanics.

### (c) Steelman

"Every design here admits cross-shard write skew; A and C just wrap their equally-unserialized foreign reads in a prettier snapshot. I refuse the one mechanism — a global min over writers — that turns any single wedged process into fleet-wide staleness, and that both rivals mis-specified into an actual corruption/stall hole. My anomalies are staleness-shaped and per-group-isolated; theirs are min-frontier-shaped and fleet-wide. For an ops story you can put in self-hosting docs — 'a sick shard hurts only itself' — that trade is correct, and the causality gap is mitigable later (e.g., embed a group-0 low-water in each commit and have serving nodes wait it out) without touching the write path."

## Design C — sequenced epochs

### (a) Strongest attack: C maximizes the blast radius of the exact failure it shares with A — and adds a mis-dated timestamp bomb and unspecified ack semantics

**1. The min-exclusion trap, node-sized, vaguer than A's.** `closedEpoch = min over nodes holding ≥1 live lease of promisedEpoch`, and §1: *"the dead node pins the min until its lease expires and evicts it."* Same question as A: what fences the evicted node's idle-in-transaction batch (epoch-E data + promise-E, all statements executed including the `FOR SHARE` generation read, waiting on `COMMIT`)? Generation bumps require the successor to take the advisory locks — held by the wedged session. If eviction is expiry-based, the late batch lands below an already-advanced frontier: identical corruption to A's, at node granularity, so **one wedged node can plant sub-frontier rows for every shard it owns at once**. C's `FOR SHARE` read is actually the raw material for a correct fence (a fencer's generation UPDATE blocks behind the batch's share lock, so fence-then-evict serializes properly) — but the design as written ties eviction to the lease TTL clock, not to a completed fence, and never states the blocking behavior. A at least wrote down a (wrong) proof; C doesn't write one.

**2. Everything-at-frontier turns any stall into a total reactivity freeze.** C's §5 "simplification": *all* served reads — writer-local, single-shard, everything — happen at the global closed frontier. So the failure A confines to cross-shard visibility (single-shard mutation reads stay hot at `lastCommitted(s)`, per A §1.3) and B confines to one group, C applies to **every read in the fleet**: one GC-paused node that owns one cold shard freezes every subscription, every RYOW, every dashboard, for up to lease-TTL, while writes keep committing invisibly ("the app froze but nothing is erroring" — C's own words). C admits this as weakness 1 but prices it as an edge case; under a correctness-and-failure-modes lens it is the *steady-state posture*: C deliberately couples the p99 of every read in the system to the min over all nodes' flush loops. Also understated in steady state: RYOW and read-after-write now wait for **every node's** promise to pass the commit's epoch (50ms idle heartbeats × K nodes), not "comparable to today's watermark wait" — today's wait is one replica's NOTIFY-driven lag; C's is a fleet-wide min including nodes that have nothing to say.

**3. The 2^53 bomb is mis-dated by ~35×.** C says ts exceeds 2^53 "within a few years." Do the arithmetic: `ts = epoch << 25`. If `epochBase` is a calendar-fixed base, ts starts around 2^57 — **over 2^53 on day one**. If the base is deployment birth, 2^53 >> 25 = 2^28 ticks × 10ms ≈ **31 days**: every missed `Number(bigint)` edge (the audit lists `write-fanout.ts:50`, `node.ts:545` — I verified `commitTs: Number(inv.newMaxTs)` in node.ts — plus `protocol.ts`, `forwarder.ts:104`, and `_creationTime`'s neighbors) passes every test, every E2E, every soak of normal length, then silently truncates in month two of production — two distinct commits comparing equal at a protocol edge, watermarks that stop advancing, RYOW waits that never satisfy. C's slice 0 is self-described as "the single most likely place this project eats an outage"; the correct statement is stronger: it is a **latently-timed** outage the test suite is structurally unable to catch unless someone writes a "start the epoch near 2^28" harness — which the design doesn't specify.

**4. Ack/durability semantics of the tick are unspecified, and batch aborts amplify.** When does a mutation ack? If pre-flush: an acked write can be lost on node crash — a regression from today's commit-means-durable, disqualifying. If post-flush (the charitable reading of "≤1 tick queueing"): then one batch abort (fencing race, PG error, one poisoned oversized row) **fails every mutation in the tick at once**, and they cannot be independently OCC-retried — serial execution means the node must re-run the whole admission queue, while their `ctx.scheduler` co-commits and component writes are entangled in the same batch. Today one commit fails alone. C never addresses tick-failure semantics at all. Adjacent unexamined stall: foreign reads execute at `readFrontierTs` against the local replica; if the local tail lags the announced frontier, the mutation blocks — and under serial execution it head-of-line-blocks *its entire shard* behind a background tailer.

Where C is genuinely strong: the encoding does preserve every scalar surface (StateVersion, `maxCommitTs` fold, `waitFor`, cursors) and uniquely *revives* the version-gap-resync optimization; exactly-once retry via intents is a real, un-matched capability; and "moving a shard = moving a lease, zero data copy" quietly deletes the resharding problem that is A's and B's worst deferred item.

### (b) Honesty of stated weaknesses

Mixed. Weaknesses 1 and 2 are honest and well-argued (the min-frontier coupling and hot-shard serial ceiling are named without flinching). Weakness 3 names the right risk but **materially understates the timeline** (2^53 in ~31 days or day one, not years) and omits the tick-abort/ack semantics entirely. The failover section's fencing story is asserted ("physically impossible, not merely improbable") with the same eviction-vs-fence conflation as A, un-proved.

### (c) Steelman

"Both rivals have the same eviction gap; mine is the only design whose flush already takes a `FOR SHARE` on the very rows a fencer must update — specify eviction as fence-then-evict and the blocking semantics fall out of Postgres row locking, no new machinery. The global-frontier read posture isn't timidity, it's the only posture under which 'everything ≤ ts is reflected' stays *literally true*, which buys back the version-gap resync, scalar RYOW, exactly-once retries, and cursors — the entire deferred roadmap. And the 2^53 edge is a grep plus a protocol-v2 test fixture that starts the clock at 2^28; a known, mechanical migration beats A's novel frontier protocol and B's vector plumbing for total correctness risk."

Partially convincing — except the everything-at-frontier read posture is a *choice*, and it's the choice that makes C's failure modes the largest of the three.

## Ranking (correctness & failure modes)

**1. Design B — 8/10.** The only design with no global-min visibility rule to mis-specify, and therefore the only one whose failure analysis survived contact: per-group frontiers are published in the commit's own fenced transaction, density is a one-writer-one-line construction fact, the wedged-writer interleaving that breaks A (and plausibly C) is a per-group stall here by inspection, and fencing rests on the session-scoped advisory lock the shipped fleet already proved. Its real costs are semantic, not protocol-safety: effect-before-cause across groups on the headline subscription feature, unbounded (and auth-touching) frozen-read staleness, and a fat, under-priced query-engine diff — all degradation-of-contract, none of them lost updates, skipped commits, or split brain. Under this lens, "weaker contract, honestly stated, structurally safe" beats "stronger contract resting on a proof with a hole."

**2. Design A — 6/10.** The most elegant architecture and the best product semantics (true prefix snapshots, unchanged client, cross-shard subscriptions done right), sunk a full tier by one thing: its central safety theorem is wrong at the live-lease-exclusion step — the exact skipped-ts-99 class it claims to have made "structurally dead" reappears via an idle-in-transaction straggler on an expired-but-unfenced lease, and the §1.2 proof sketch papers over it by conflating expiry with promotion. The hole is fixable (fence-then-evict), but the fix converts A's advertised "dead shard drops out of the min" into the fleet-wide stall it positioned itself against, and the 5s-RYOW-timeout path turns that stall into client-visible RYOW violations. A design whose proof I can falsify with one interleaving scores below a design whose weaknesses are on the label.

**3. Design C — 5/10.** Intellectually the richest — the encoding trick is real, and intents/lease-move-resharding are capabilities the others can't match — but it accumulates the most independent correctness liabilities: the same eviction-vs-fence gap as A, stated more vaguely; a read posture that maximizes the blast radius of any single wedged node to *all reads everywhere* (strictly worse than A's cross-shard-only and B's one-group coupling); a front-loaded, everything-touching timestamp migration whose 2^53 failure window is ~31 days, not "a few years," and is invisible to any normal test suite; and wholly unspecified tick ack/abort semantics on which "commit = durable" — the oldest invariant in the engine — currently hangs. Each is individually fixable; a design that needs four fixes to its safety story before slice 1 is the riskiest thing on this table, and "riskiest" is the whole question under my lens.

**Cross-cutting verdict for the parent:** the decisive discriminator this review surfaced is not throughput or DX — it is that **any `min-over-live-X` visibility rule is unsound unless eviction from the min is fencing-first**, because in the shipped lease machinery expiry is a clock and fencing is a lock transfer, and only session death connects them. B avoids the rule entirely; A and C must both respecify eviction (bump-epoch/generation serialized against the in-flight commit's own row locks, then exclude) and re-price their failure-mode claims with the resulting TTL-bounded stall before either is safe to build.
