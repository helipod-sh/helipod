# E4 — Lunora at the source level, browser realities, and the rest of the field

Scope: (1) a source-level audit of Lunora's durable-offline-outbox stack — the `clientId + clientSeq` watermark protocol, its atomicity, `lastMutationId` reconciliation, the offline queue's durability + drain, and cursor/bookmark resume — verified against the clone at `.reference/lunora` (HEAD `2e8df7b`, 2025-06-22; alpha — line numbers valid for that commit only). This supersedes the docs-level treatment in [`docs/dev/research/lunora.md`](../lunora.md) §5/§10 wherever they disagree. (2) The browser environment any web outbox must survive (eviction, quotas, multi-tab, Safari, background sync) — web-sourced, URLs cited. (3) Linear, Firestore, and the CRDT school as calibration points. File:line for code claims (`.reference/lunora/<path>:<line>` for the clone); URLs for web claims; uncertainty flagged inline. Never copy reference code — study and describe.

Headline: **Lunora does not ship one outbox — it ships two coexisting idempotency schemes**, and the earlier docs-derived summary ("watermarks give idempotency", lunora.md:62) conflated them. (A) The **built-in `OfflineQueue`** (plain mutations): per-call `(identity, mutationId)` dedup rows + a `commitCursor` echo for optimistic-overlay drop — concurrent-safe, order-agnostic. (B) The **custom-mutator watermark protocol** (`@lunora/db` local-first path): per-`(identity, clientId)` monotonic `clientSeq`, serialized per client, `lastMutationId` echoes. The watermark path is strictly serialized by construction (a client-side FIFO push chain exists *because* the server rejects any gap); the plain path stays concurrent. Second headline: **the watermark advance IS atomic with the data write on the primary path** — `commitMutationBookkeeping` runs inside the mutation's transaction — and the non-atomic fallback path is deliberately designed to self-heal by replay, never by skip. Third: the browser is actively hostile to "durable" — best-effort IndexedDB is evictable as a whole-origin unit, Safari deletes it after 7 days of non-interaction, Background Sync doesn't exist off Chromium — so any spec claim of durability must be qualified to "as durable as the origin's storage, plus server-side idempotency as the real safety net." That last clause is the one invariant every system in this file (Lunora, Linear, Firestore) independently converged on.

## Part 1 — Lunora, at the source level

### 1.1 The two schemes, and which one the spec's head-to-head actually bound

| | Built-in `OfflineQueue` (plain mutations) | Custom-mutator watermark (`@lunora/db`) |
|---|---|---|
| Client tag | `x-lunora-mutation-id` = `${clientId}:${counter}` (`lunora-client.ts:2603, types.ts:152`) | `x-lunora-client-id` + monotonic `x-lunora-client-seq` (`shard-do.ts:1920–1921`) |
| Server guard | `(identity, mutationId)` dedup row in `__idempotency`, cached result replayed (`shard-do.ts:2001–2005, 3299–3313`) | `(identity, client_id)` high-watermark in `__client_watermark` (`ctx-db-client-watermark.ts:50–60`) |
| Ordering | none — writes are independent, concurrent | strict per-client serialization; gaps rejected |
| Overlay drop signal | echoed `commitCursor` (CDC high-watermark) (`shard-do.ts:3514–3533`) | echoed `lastMutationId` (`shard-do.ts:3527`) |
| Durable client store | own IndexedDB DB `lunora-outbox` (`persistence.ts:57`) | TanStack `@tanstack/offline-transactions` executor (`db/src/define-collections.ts:264–268`) |

A push without the client-seq headers "rides the legacy idempotency dedup" (`shard-do.ts:1918–1919`); the two schemes share the dispatch path and the `__idempotency` table backstops both. Any Stackbase position paper comparing against "the Lunora watermark" must say which half it means — the serialized watermark buys total per-client order (and pays for it with head-of-line blocking, §1.5), the plain scheme buys concurrency (and gives up cross-mutation ordering).

### 1.2 Client-side tagging

- `clientId`: constructor `options.clientId ?? "client-" + nextId()` (`lunora-client.ts:753`) — **ephemeral per session by default**. `nextId()` is `crypto.randomUUID` with a crypto/timestamp/counter fallback (`offline-queue.ts:70–84`), and MUST be globally unique because an anonymous client's watermark is keyed `("", clientId)` (`offline-queue.ts:60–69`, `ctx-db-client-watermark.ts:15–22`). A **stable** clientId is only present when the app supplies one; the `@lunora/db` path is documented to "persist a stable id alongside the outbox and pass it here" (`client/src/types.ts:250–257`) — the persistence itself lives app-side/scaffold-side, not in `@lunora/client`.
- `clientSeq` is never minted by the transport. `callMutator(path, args, {clientSeq})` validates it is a positive integer or absent — an accidental `0` would be `<=` the initial watermark and be **acked as an already-applied replay without running the handler**, a silent dropped write that the guard exists to prevent (`lunora-client.ts:887–903`). Seq assignment is owned by `bindMutators` (§1.5).

### 1.3 Server-side watermark: storage, classification, responses

Storage — `__client_watermark (identity TEXT, client_id TEXT, last_mutation_id INTEGER, PRIMARY KEY (identity, client_id))` (`ctx-db-client-watermark.ts:53–58`). Keyed by *authenticated* identity precisely because `clientId` is client-supplied and unauthenticated: without identity in the key, one user could reuse another's `clientId` and suppress their sequence (`ctx-db-client-watermark.ts:6–12`). Reads default to `0` for an unseen pair (`:69–76`); the advance is a `MAX(last_mutation_id, excluded.…)` upsert so the column is monotonic even under a buggy out-of-order advance (`:89–95`).

Classification — before the handler runs, `classifyClientMutation()` reads the watermark and returns exactly one of (`shard-do.ts:3394–3432`):

- `"already"` (`seq <= watermark`) → ack **without re-running**: `200 { lastMutationId: watermark, result: null }` so the client can drop its pending overlay (`shard-do.ts:3448–3453`).
- `"next"` (`seq == watermark + 1`) → run the authoritative impl and advance the watermark.
- `"gap"` (`seq > watermark + 1`) → `409 { error: { code: "OUT_OF_ORDER", expectedMutationId } }`, halt: "the client must resend from `watermark + 1`; never apply out of order" (`shard-do.ts:3456–3468`).

A missing `__client_watermark` table is self-healed by re-running the DDL inline rather than silently downgrading to the legacy path (`shard-do.ts:3411–3423`).

### 1.4 Where the watermark commits — the atomicity story, precisely

Three layers, from strongest to backstop:

1. **Primary (atomic).** The generated `handleRpc` mutation branch calls `commitMutationBookkeeping(result)` *inside the mutation's transaction*, after the user handler resolves but before commit: the data writes, the `__idempotency` dedup row, and the watermark advance "land in one atomic commit: a crash can't leave the writes durable without the replay guard … nor without the watermark" (`shard-do.ts:3535–3556`). The advance is `strict` there — a watermark write failure **rolls the whole mutation back** rather than committing writes with no advanced watermark (`shard-do.ts:3549–3552, 3596–3603`).
2. **Fallback (non-atomic, self-healing).** Dispatch paths that aren't transaction-wrapped run `recordPostDispatchBookkeeping` after the handler returns — skipped when the flag says the transactional path already did it (`shard-do.ts:3566–3576`; the dispatch-site comment at `shard-do.ts:2015–2030` documents the crash contract: writes commit → crash before advance → the client's unacked replay re-classifies as `"next"` → re-runs, but the `__idempotency` row dedups the actual write → re-advances. "Only ever *replaying*, never skipping or double-applying." The stale module-level comment at `ctx-db-client-watermark.ts:78–88` describes only this fallback; the dispatch-flag handshake at `shard-do.ts:1922–1926` is the ground truth.)
3. **Crash-window repair on the cached path.** A replay that hits the idempotency cache while classified `"next"` (handler committed, watermark advance lost to a crash between the two — only reachable via path 2) re-advances and echoes `lastMutationId` exactly as the fresh path does — "otherwise the cached branch returns a bare result with a stale watermark and the client reports every later seq as a gap forever" (`shard-do.ts:3471–3492`).

Takeaway for Stackbase: our transactor gives us layer 1 for free — the watermark/dedup row is just another write in the OCC commit. Lunora needed three layers because DO SQLite auto-commits per statement outside the wrapped path; we should need one, plus the cached-result echo.

### 1.5 `lastMutationId` reconciliation and the client seq generator (`@lunora/db`)

- **Optimistic overlay lifecycle.** `bindMutators` opens a TanStack optimistic transaction: the mutator's `apply` body writes predicted rows locally, `mutationFn` pushes the authoritative write via `callMutator` (`db/src/define-mutators.ts:162–190`). Overlay drop is gated on a monotonic **checkpoint registry**: `awaitMutationId(seq)` resolves when any frame echoes `lastMutationId >= seq` (`db/src/collection-options.ts:15–90`), fed by `data`/`delta` frames and poke checkpoints (`lunora-client.ts:3799–3809, 3904–3912`). Holding the overlay until the *synced row lands* (not merely until the ack) is what prevents the flash-of-disappearing-row (`collection-options.ts:53–58`).
- **Seq generator seeded from the server.** The in-memory counter is `max(counter, confirmedMutationWatermark(shard)) + 1` on every issue (`define-mutators.ts:115–119`), where `confirmedMutationWatermark` is the highest echoed watermark (`lunora-client.ts:855–864, 917–919`). Reason: a reload resets the counter while the server keeps a durable watermark; reissuing an already-applied seq would be *swallowed as a replay, silently dropping the write* (`lunora-client.ts:856–860`).
- **The `applied` verdict + reissue loop.** `callMutator` returns `applied: false` when the echo shows the push was acked as a replay; `bindMutators` then **reissues the same args above the now-known watermark** (bounded at 32 attempts) instead of mistaking the benign ack for a confirmed write (`lunora-client.ts:872–877, 921–926`; `define-mutators.ts:105–149`). The doc comment argues a fresh session's first stale push "provably cannot be an honest replay" (`define-mutators.ts:89–94`) — true only while `clientId` is fresh per session; with a persisted stable `clientId` the reissue re-sends args whose original *did* apply pre-reload, and the idempotency row is what keeps that from double-applying. Flag: this coupling (stable clientId ⇒ reissue safety leans on `__idempotency`) is implicit, not documented.
- **The FIFO cost.** Because a gap is a hard 409, "pushes are serialized per binding: each push waits for the previous one's ack and assigns its `clientSeq` inside the critical section" (`define-mutators.ts:96–103, 121–160`). One RTT per mutation, head-of-line blocking under latency. A failed mutation never advances the server watermark, so the next push reclaims the same `watermark + 1` — a permanently-rejected write can't wedge the chain (`define-mutators.ts:101–103`).

### 1.6 The built-in offline queue: durability + drain

**Durability.** Bounded in-memory FIFO (default `maxItems: 1000`) mirrored to a `PersistenceAdapter` on enqueue (`offline-queue.ts:117–191`). The default adapter is IndexedDB, DB `lunora-outbox`, store `offline-mutations`, autoincrement primary key so `load()` preserves enqueue order regardless of string ids, unique secondary index on `id` for `remove()` (`persistence.ts:57–60, 83–160`). Auto-probed only when `indexedDB` exists — SSR/Node/RN stay in-memory (`persistence.ts:180–194`); it is deliberately a **separate IndexedDB database** from the read cache because IndexedDB versioning is per-database and sharing one DB forced two independently-toggleable adapters to share a version constant ("they didn't, which threw `VersionError`") (`persistence.ts:51–56`). Persisted record carries `{id, functionPath, args, identity, shardKey, version}`.

**Failure honesty.** Persistence `append` failures are swallowed but reported (`onPersistenceError` or `console.warn`, `offline-queue.ts:86–103, 156–167`). Overflow evicts the *oldest* entry, un-persists it, rejects with `OFFLINE_QUEUE_OVERFLOW`, and additionally fires an observer because a hydrated entry's `reject` is a no-op — "without this an overflow eviction would silently drop a durable write" (`offline-queue.ts:169–187`). `clear()` (client close) rejects awaiting callers `CLIENT_CLOSED` but **leaves durable storage intact** — "closing a tab must not discard writes a future session will restore" (`offline-queue.ts:294–309`).

**Hydrate.** `hydrate()` reloads persisted records FIFO with no-op resolvers (the original awaiter is gone), dedups by id, and applies a **version gate**: a record stamped under a different `persistenceVersion` is dropped + purged rather than replayed against a changed schema (`offline-queue.ts:201–247`; single shared predicate `persisted-version.ts:12` — gating is off unless the app opts in, and turning it on invalidates all legacy unstamped records). Returns the distinct shard keys so the client can open exactly those sockets to trigger flushes (`lunora-client.ts:2676–2686`).

**Who hydrates: the multi-tab leader.** Hydration runs only in the tab holding the Web Lock `lunora:outbox-leader:${url}`; the leader holds the lock for its lifetime (an unresolved promise) so on close the next waiting tab takes over; no-Web-Locks environments hydrate unconditionally (single-context) (`lunora-client.ts:2688–2734`). Rationale spelled out: without coordination every tab replays the same writes — "correct only because the server dedups by idempotency key, but wasteful + racy" (`:2690–2692`). Note what is *not* coordinated: live mutations in follower tabs still send directly; the lock scopes only durable-queue hydration.

**Enqueue triggers + identity binding.** A mutation that can't go on the wire (offline / mid-reconnect) has already applied its optimistic update; `enqueueOfflineMutation` binds the **issuing identity fingerprint at enqueue time** and persists it with the record (`lunora-client.ts:2571–2669`). The queue id reuses the call's idempotency key "so the replay carries the same `x-lunora-mutation-id` the server dedups on" (`:2633–2635`).

**Drain.** Socket (re)connect for a shard → `flushOfflineQueue(shardKey)` (`lunora-client.ts:3392, 4153–4222`):

1. Predicate-drain only that shard's entries, preserving FIFO for the rest (`offline-queue.ts:255–277`).
2. **Identity gate per entry** against one snapshot: a write stamped under a different identity is rejected `OFFLINE_IDENTITY_CHANGED`, never replayed under the wrong user; hydrated entries fall back to the persisted stamp (`lunora-client.ts:4161–4179, 4249–4285`). A credential *change* also drains + rejects everything up front (`:4103–4118`) — Lunora's answer to "what does the outbox do on logout" is **discard, loudly**, not replay-later. A label-only change (user id resolving after token set) re-stamps instead of discarding (`:4120–4134`).
3. **Encodability triage**: args that can't wire-encode are deterministically unreplayable — settle terminal now, or the codec throw would be classified transient and re-queue forever, "a silent hang" (`:4224–4247`).
4. Replay: one write rides the single-call path; N > 1 coalesce into `/_lunora/rpc-batch` chunks capped at `MAX_BATCH_ENTRIES`, chunks sequential to preserve FIFO (`:4187–4217`). **Coded error = server verdict → drop terminally (poison-message rule); codeless error = transport/transient → stop the flush, requeue this and every unreplayed write in order** (`:4296–4300, 4302–4311`). Requeue does not re-persist — the records were never unpersisted (`offline-queue.ts:279–292`).
5. Success: un-persist, confirm the per-call optimistic layer against the echoed `commitCursor` *before* resolving the awaiter (`:4287–4293`).

Every terminal verdict also fires a `MutationSettledEvent` observer with `hadAwaiter` so post-reload replays (whose promise died with the old page) are still observable by app code (`offline-queue.ts:16–23`, `lunora-client.ts:205–215` region).

### 1.7 The `@lunora/db` outbox: delegate to TanStack, keep the verdict split

When the TanStack executor is wired, the built-in queue's implicit IndexedDB default is suppressed (single durable write path — "the built-in queue must stay in memory rather than persist a second, never-flushed copy", `persistence.ts:172–176`) and offline writes delegate to `outbox.enqueue({clientId, mutationId, idempotencyKey: "clientId:counter", identity, …})`, returning optimistically — confirmation rides the synced view (`lunora-client.ts:2577–2623`). The executor (`startOfflineExecutor`) owns durable storage, FIFO processing, retries, and its own leader election (`db/src/define-collections.ts:264–277`; TanStack docs: outbox pattern, IndexedDB with localStorage fallback, multi-tab via `navigator.locks` + BroadcastChannel — https://www.npmjs.com/package/@tanstack/offline-transactions). Lunora contributes the retry *policy*: `runOutboxMutation` maps any error carrying a machine-readable `code` to `NonRetriableError` (server verdict → roll the optimistic insert back) and rethrows codeless errors as transient for durable replay — keying on `instanceof TypeError` alone would wrongly drop 5xx/non-JSON infra blips (`db/src/internals.ts:202–233`). And it deliberately **distrusts `navigator.onLine`**: the online detector always answers `true` and a 1s tick nudges the drain, because stuck `onLine` (observed under Playwright/Firefox `setOffline`) "would freeze the outbox" (`db/src/internals.ts:7, 235–272`).

### 1.8 Resume: CDC cursor + epoch (and the unrelated D1 bookmark)

Two different "bookmarks" — do not conflate:

- **CDC cursor/epoch resume** (the reconnect story). Every `data`/`delta` frame stamps the `__cdc_log` high-watermark as `cursor` plus a shard `epoch` (`shard-do.ts:3175–3195`). On resubscribe with `sinceSeq`, `evaluateResume` returns `resumable: true` only if: CDC is on, the client's epoch matches (else a reset/recycled-DO timeline fork → re-snapshot), `sinceSeq <= cursor` (a client claiming more than the shard holds means a rollback/PITR → re-snapshot), the retention floor covers `(sinceSeq, cursor]` (compaction gap → re-snapshot), the read-set is known (unknown deps → re-snapshot), the missed-change scan is under its cap, and no table in the query's read-set changed since `sinceSeq` (`shard-do.ts:3197–3288`). Resumable → a lightweight `resume` frame instead of re-shipping the snapshot. This is a **conservative prove-or-reseed design**: six independent conditions each force the safe fallback. Our MVCC log's `load_documents(tsRange)` is the same substrate with strictly better properties (real snapshots, no epoch improvisation needed if the deployment id is the epoch).
- **D1 bookmark** (`x-d1-bookmark`) is the Cloudflare D1 Sessions read-your-writes token for the *global* eventually-consistent tier, threaded through request/response headers (`shard-do.ts:1908–1912`; `client/src/bookmark.ts:4–17`). Irrelevant to the outbox except as naming noise.

### 1.9 What E4 would carry into a Stackbase position

Adopt: (a) the **three-way classify before the handler** (`already`/`next`/`gap`) with an explicit `OUT_OF_ORDER` + `expected` echo — it makes the client's resend algorithm trivial and testable; (b) **identity-scoped keys** for both watermark and dedup rows — client-supplied ids are never trusted alone; (c) the **coded-verdict vs codeless-transient split** as the single retry policy, plus the encodability triage; (d) **identity stamped at enqueue, gated at flush, discarded loudly on auth change**; (e) the **Web Lock outbox-leader** with server-side dedup as the correctness backstop (locks are an optimization, not the safety); (f) the **overlay-drop gate on `lastMutationId`/commit-ts echoed through the sync stream**, not on the mutation response alone. Avoid/weigh: the per-client FIFO chain's head-of-line blocking is the price of total order — Stackbase's plain-mutation path (like Lunora's scheme A) should stay concurrent, with ordering opt-in; and the ephemeral-by-default `clientId` quietly orphans server watermark rows (no GC shown for `__client_watermark` in the clone — uncertainty: may exist elsewhere; `__idempotency` has throttled GC, `shard-do.ts:3349–3354`).

## Part 2 — Browser realities the design must survive

### 2.1 IndexedDB is evictable by default; "persistent" is a mode you must request

All origins start in **best-effort** mode: under storage pressure the browser evicts the **entire origin's data as one unit** (all storage APIs at once, LRU order), skipping origins granted persistence via `navigator.storage.persist()` ([MDN: Storage quotas and eviction criteria](https://developer.mozilla.org/en-US/docs/Web/API/Storage_API/Storage_quotas_and_eviction_criteria)). Consequences for an outbox: (a) there is no partial eviction — the queue and the read cache die together; (b) `persist()` is a *request*: Firefox prompts the user, Chrome and Safari silently grant/deny based on interaction heuristics ([MDN Storage API](https://developer.mozilla.org/en-US/docs/Web/API/Storage_API); [Dexie StorageManager guide](https://dexie.org/docs/StorageManager)) — a design cannot *assume* persistence was granted and must treat the grant as advisory; (c) quotas are generous on Chromium (best-effort up to ~60% of disk per origin) and Firefox (min(10% of disk, 10 GiB) best-effort; ~50% persistent), so an outbox will realistically die by *eviction or Safari policy*, not by quota — but writes can still throw `QuotaExceededError` and the adapter must surface that as a persistence failure, not a mutation failure (Lunora's swallowed-but-reported pattern, §1.6). `navigator.storage.estimate()` gives usage/quota for a proactive advisory (the "warn at 10% of the ceiling" pattern lunora.md:133 catalogs).

### 2.2 Safari is the adversarial case

- **7-day script-writable-storage cap**: with ITP on, Safari deletes ALL script-writable storage (IndexedDB, localStorage, service-worker registrations) for a site after seven days of Safari use without user interaction on that site ([WebKit policy, via Search Engine Land's analysis](https://searchengineland.com/what-safaris-7-day-cap-on-script-writeable-storage-means-for-pwa-developers-332519); [WebKit storage-policy update](https://webkit.org/blog/14403/updates-to-storage-policy/)). Home-screen web apps keep a separate days-of-use counter and are exempt in practice. A durable outbox on Safari is therefore durable for **at most 7 days of user absence** — a queued write older than that may simply vanish. Design consequence: the outbox contract must be time-bounded ("writes survive reloads; they do not survive weeks of absence on Safari") and the server-side idempotency window sized accordingly.
- **Proactive eviction** applies to Safari even absent storage pressure ([MDN eviction criteria](https://developer.mozilla.org/en-US/docs/Web/API/Storage_API/Storage_quotas_and_eviction_criteria)).
- **Private browsing**: data is wiped at session end in all engines; historical Safari private mode threw on IndexedDB open (modern Safari allows it but ephemerally — uncertainty: exact current Safari private-mode IndexedDB behavior varies by version; treat "adapter constructor/open may throw or store may be ephemeral" as the contract). Lunora's answer — adapter probe + graceful in-memory fallback (`persistence.ts:180–194`) — is the right shape.
- **iOS PWA storage** can additionally be cleared by the OS after weeks of non-use ([vinova summary of iOS PWA limits](https://vinova.sg/navigating-safari-ios-pwa-limitations/)).

### 2.3 Multi-tab: exactly one drainer, and the server must not care

Three coordination primitives, one pattern: **Web Locks** (`navigator.locks.request(name, cb)` — held-for-lifetime lock = leader election with automatic failover on tab close/crash; this is what both Lunora (`lunora-client.ts:2698–2734`) and TanStack offline-transactions use), **BroadcastChannel** (cheap same-origin pub/sub for "I committed, refresh" signals — no ownership semantics), **SharedWorker** (a real single execution context owning the socket + drain — strongest model, but unavailable on Chrome for Android and historically flaky on Safari, so it can only ever be a progressive enhancement). The load-bearing insight from Lunora: leadership is an *efficiency* mechanism; **correctness comes from server-side idempotency keys** — if two tabs drain the same persisted queue, the second replay dedups server-side (`lunora-client.ts:2690–2692`). Any design that is only correct when leader election works is wrong: locks can be unavailable (older WebViews, RN), and IndexedDB itself is shared mutable state between tabs regardless. Secondary hazards: per-tab in-memory `clientSeq` counters + a shared `clientId` = instant `OUT_OF_ORDER` storms (Lunora's watermark path implicitly requires the seq generator to be single-context or watermark-reseeded, §1.5); and hydrate-time dedup by record id must tolerate a second tab having already replayed+removed a record.

### 2.4 Service-worker Background Sync: what it buys, what it doesn't

One-shot Background Sync (`registration.sync.register(tag)`) fires a `sync` event in the service worker when connectivity returns — **even if the page is closed** — with automatic retry/backoff on a browser-controlled schedule ([MDN Background Synchronization API](https://developer.mozilla.org/en-US/docs/Web/API/Background_Synchronization_API)). What it buys: drain-after-tab-close on Chromium; the one capability no in-page mechanism can replicate. What it doesn't: **Firefox and Safari have never shipped it** (~76% global support, Chromium-only — [caniuse](https://caniuse.com/background-sync); [testmuai support survey](https://www.testmuai.com/learning-hub/background-sync-browser-support/)); the sync event has a tight execution budget; firing timing is browser-discretionary; and the SW context can't touch the page's in-memory state, so the queue must live in IndexedDB in a format the SW can replay independently (which also means the replay path — auth token included — must be readable from the SW, an auth-storage design constraint). Periodic Background Sync is even narrower (Chromium + installed-PWA + engagement-gated). Verdict for a v1 design: treat Background Sync as a progressive enhancement on the drain trigger, never as the durability story; the portable baseline is exactly Lunora's — IndexedDB queue + drain on next page load/reconnect + interval nudge (§1.7's 1s tick, distrusting `navigator.onLine`).

## Part 3 — The rest of the field

### 3.1 Linear: transaction queue + undo (writeups)

From the CTO-endorsed reverse-engineering ([wzhudev/reverse-linear-sync-engine](https://github.com/wzhudev/reverse-linear-sync-engine)) and Artman's talks ([Scaling the Linear Sync Engine](https://linear.app/now/scaling-the-linear-sync-engine); [fujimon summary](https://www.fujimon.com/blog/linear-sync-engine)):

- **Typed transactions, not raw args**: six transaction classes (Creation/Updating/Deletion/Archival/Unarchive/Local) serialized to an IndexedDB `__transactions` table *before* sending; queue states `created → queued → executing`, plus `persistedTransactionsEnqueue` for post-restart restore. Batching: same-`batchIndex` transactions merge into one GraphQL request with aliased sub-mutations.
- **Ack ≠ complete**: the server returns `lastSyncId` per mutation; a transaction parks in `completedButUnsyncedTransactions` until the **delta packet with the matching `lastSyncId` arrives on the sync stream** — the same "confirmation rides the synced view, not the response" convergence Lunora reaches with `lastMutationId` gates (§1.5) and our E1 reaches with the commit-ts transition gate. Three systems, one shape: **the overlay drops when the read path catches up to the write, never merely when the RPC returns.**
- **Rejection → rollback**: a server rejection still calls `transactionCompleted` with an error and triggers the transaction's `rollback()` to undo client-side state — reject-and-rollback, not merge (server-authoritative, like Lunora and unlike CRDTs).
- **Undo is transaction-shaped**: transactions know how to serialize/replay themselves (`fromSerializedData` replays against in-memory models); undo leverages reversing recorded transaction changes rather than snapshotting whole documents. Design signal: if the outbox entry is expressive enough to replay, it is nearly expressive enough to *reverse* — an argument for structured outbox entries over opaque `{path, args}` blobs, though Linear can do this only because its writes are model-property deltas (LWW per property), not arbitrary mutation functions. For Stackbase's arbitrary-TS mutations, client-side reversal is not generally derivable — undo would be an app-level compensating mutation, which is worth stating explicitly in the verdict rather than implying Linear-style undo is reachable.
- Offline catch-up: handshake compares local vs server `lastSyncId`, fetches missing delta packets, else full re-bootstrap — bookmark-resume over an ordered log again. Conflicts: LWW per property; Artman notes real conflicts are rare in practice; CRDTs adopted only for issue-description text ([bytemash rabbit-hole writeup](https://bytemash.net/posts/i-went-down-the-linear-rabbit-hole/)).

### 3.2 Firestore: the offline-persistence contract of record

The mainstream baseline ([Firestore: Access data offline](https://firebase.google.com/docs/firestore/manage-data/enable-offline)): writes while offline are queued (IndexedDB on web when persistence is enabled), listeners fire immediately with latency-compensated local data flagged `fromCache`/`hasPendingWrites` in snapshot metadata, and on reconnect the SDK syncs the queue — "for multiple changes to the same document, it's last write wins." No cross-document atomicity for the replayed queue; transactions *fail* offline rather than queue (they require server round-trips). Multi-tab is an explicit opt-in (`persistentMultipleTabManager()`), historically fragile ("multiple tabs open" errors — [firebase-js-sdk#720](https://github.com/firebase/firebase-js-sdk/issues/720)). Cache is bounded (default 100 MB per the current docs, GC of older unused docs; the older documented default was 40 MB). The honest failure mode: Firestore "was not designed as an offline-first database" — the local cache has **no indexes**, so offline queries unpack and scan every cached document in the collection; extended offline accumulation degrades reads ([Kerpelman, "Why is my Cloud Firestore query slow?"](https://medium.com/firebase-developers/why-is-my-cloud-firestore-query-slow-e081fb8e55dd)). There are also long-standing gaps our design should learn from as *API* lessons: no first-class way to enumerate/observe the pending-write queue ([firebase-js-sdk#3661](https://github.com/firebase/firebase-js-sdk/issues/3661)), and a rebooted device's queued transactions may not flush until a new write nudges the pipeline ([firebase-android-sdk#3221](https://github.com/firebase/firebase-android-sdk/issues/3221)). Lunora's pending-count observer + settled-event observer (§1.6) are direct answers to #3661.

### 3.3 The CRDT school (Automerge), and why it doesn't fit server-authoritative mutations

CRDTs solve a different problem: **merge concurrent replicas without any authority**. Every op must commute; "conflict" is defined away by construction. That is structurally incompatible with Stackbase/Convex-style mutations, whose entire value is a serializable transaction that can *reject* (validators, invariants, authz) — three specific mismatches, all visible in Convex's own Automerge exploration ([Going local-first with Automerge and Convex](https://stack.convex.dev/automerge-and-convex)): (1) **invariants**: CRDT merges guarantee convergence, not validity — nothing keeps two independently-valid edits from merging into an invalid whole, and cross-document consistency has no central authority to enforce it; (2) **server-side auditing**: to validate change *content* the server must run the CRDT engine itself, else changes are opaque blobs it can only accept ("you lose the ability to audit the content of each change"); the suggested workaround — treat increments as *suggestions* until compacted into a validated snapshot — is exactly a reinvention of server-authoritative apply; (3) **the outbox reduces to the document**: in Automerge the "queue" is the local doc's unsent change history, idempotent by op identity — elegant, but it buys convergence only for data shaped as a CRDT document, not for `ctx.db` transactions with arbitrary read-write logic. Where the school *is* right: op-identity-based idempotency (every op globally identified — the same idea as `clientId:seq`), and per-field/per-datatype merge as an opt-in for genuinely collaborative *text* — which is precisely the boundary Linear drew (§3.1). The verdict frame: server-authoritative outbox for mutations, CRDT as a possible future *field type*, never as the sync substrate.

## The environmental-hazards checklist

Every position paper must answer, for each numbered hazard, what happens and where the invariant lives (client, server, or both):

1. **Whole-origin eviction** — best-effort IndexedDB evicted as one LRU unit (queue + read cache + everything). What is lost, what does the user see, and does the server's idempotency window make a partially-replayed-then-evicted queue safe? (§2.1)
2. **Safari 7-day wipe** — queued writes older than 7 days of site non-interaction vanish on Safari. Is the outbox contract explicitly time-bounded? Is there an age advisory before the cliff? (§2.2)
3. **`persist()` denied silently** — Chrome/Safari grant/deny with no prompt; the design must degrade to best-effort without behavior change. (§2.1)
4. **`QuotaExceededError` mid-append** — a persistence failure must not fail (or silently un-durable-ize) the mutation; it must be observable (Lunora: `onPersistenceError` + warn, §1.6).
5. **Private mode / no IndexedDB at all** — adapter open may throw or be session-ephemeral; graceful in-memory fallback with the *same* API contract (§1.6, §2.2).
6. **Two tabs, one queue** — who hydrates, who drains, and is the system still correct when leader election is unavailable or misfires? (Answer must be: server-side idempotency, locks are an optimization. §1.6, §2.3)
7. **Tab closed mid-drain / killed mid-flush** — a write sent-but-unacked at page death: replay must be safe (idempotency key survives with the record), and the next session/leader must resume FIFO. (§1.6 requeue-without-unpersist; §1.4 layer 2.)
8. **Reload resets in-memory counters while server state persists** — any per-client sequence must be reseeded from a server echo or the first push silently swallows as a replay (Lunora's seed-from-watermark + `applied:false` reissue, §1.5).
9. **Auth identity changes with writes queued** — replaying under the new identity is a privilege bug; the record must carry its issuing identity, be gated at flush, and be discarded *loudly* (observer/UI), not silently (§1.6).
10. **Schema/app version changes under persisted writes** — a write minted against v1 replayed against v2: version-stamp records and drop-with-verdict on mismatch (Lunora `persistenceVersion`, §1.6) — or define a migration story; silence is the only wrong answer.
11. **Poison writes** — deterministically-failing entries (server verdict, or un-encodable args) must terminal-fail immediately, never block the FIFO or retry forever (coded-vs-codeless split + encodability triage, §1.6; TanStack `NonRetriableError`, §1.7).
12. **Queue overflow** — a bounded queue must evict *observably* (`OFFLINE_QUEUE_OVERFLOW` + settled-event even for awaiter-less hydrated records, §1.6); an unbounded one must answer Firestore's local-scan degradation (§3.2).
13. **`navigator.onLine` lies** — drain triggering must not gate on it; always-attempt + backoff + interval nudge (§1.7), reconnect with jittered exponential backoff (`reconnect.ts:17–43`).
14. **No Background Sync off Chromium** — drain-after-tab-close is a Chromium-only enhancement; the portable contract is drain-on-next-visit (§2.4). If a SW drain path exists, the queue format + auth must be SW-readable.
15. **Server log/timeline resets under a resuming client** — resume tokens must be provably same-timeline (epoch/deployment-id) and within retention, else force re-seed; a client claiming a future cursor means rollback — re-seed (Lunora's six-condition `evaluateResume`, §1.8).
16. **Ack received, sync stream not yet caught up** — the optimistic overlay must drop on the *synced view* reaching the write (commit-ts / `lastMutationId` gate), or the UI flashes; three independent systems converged on this (§1.5, §3.1, E1's ts-gate).
