# E2 — Replicache + Zero, source-level: how the one production-proven browser outbox actually works, and what its own authors walked back

Evidence agent E2 in the durable-offline-outbox workflow. Mandate: read the actual source of Replicache and Zero (Rocicorp's `mono`, cloned at `.reference/mono`, read 2025-11-04) and extract the mechanics of their pending-mutation persistence — not the marketing description of it. Code claims cite `.reference/mono/<path>:<line>`; doc claims cite the in-repo `replicache-doc` markdown (which is the published replicache.dev docs) or a live URL. Where I am reasoning past what I read, that is flagged. No reference code is copied into our tree (`.reference/README.md` rule); everything below is description.

The headline finding, up front: **Replicache's outbox is not a queue data structure — it is the client database itself.** Pending mutations *are* commits in a content-addressed DAG, and "the outbox" is a derived view (walk from the head to the last server snapshot, collect the local commits). Deduplication and ordering are not client concerns at all: they hang entirely on one server-side integer per client (`lastMutationID`), stored transactionally with the data. And the second finding, just as important: **Zero — the same team, second system — kept every bit of that machinery and then *turned off* offline writes.** When Zero's connection state is `disconnected`/`error`/`closed`, the mutator proxy doesn't even run the local mutation (`.reference/mono/packages/zero-client/src/client/mutator-proxy.ts:84-121`). The most experienced builders of browser outboxes decided a durable offline outbox was not worth its product cost for their v1. That is evidence our research has to weigh, not dismiss.

## 1. The pending-mutation log: commits in a DAG, not rows in a queue

### 1.1 What a pending mutation IS

A local mutation in Replicache is a **commit** whose meta is `LocalMetaDD31`:

```
{ type, basisHash, mutationID, mutatorName, mutatorArgsJSON,
  originalHash, timestamp, clientID, baseSnapshotHash }
```

(`.reference/mono/packages/replicache/src/db/commit.ts:257-267`). The load-bearing fields for an outbox:

- **`mutatorName` + `mutatorArgsJSON`** — the mutation is stored as *intent* (name + args), never as a computed write-set. This is what makes rebase possible: the client can re-run the intent against new state.
- **`mutationID`** — a per-`clientID` dense counter (1, 2, 3, …). Assigned at commit time as `basis.lastMutationID(clientID) + 1`; rebase asserts the invariant (`.reference/mono/packages/replicache/src/db/rebase.ts:64-72` throws `Inconsistent mutation ID` if the replay's next-ID doesn't equal the recorded one).
- **`originalHash`** — non-null iff this commit is a *replay* of an earlier commit (points at the original), so the system can distinguish first-run from rebase.
- **`basisHash` / `baseSnapshotHash`** — the DAG linkage that makes "pending" a derived property.

"The pending queue" is computed, not stored: walk from the main head back to the base snapshot and collect local commits — `localMutations(mainHeadHash, dagRead)` filters the chain (`.reference/mono/packages/replicache/src/db/commit.ts:112-127`); the public `experimentalPendingMutations` API is exactly this walk, reversed to oldest-first (`.reference/mono/packages/replicache/src/pending-mutations.ts:16-29`). Acknowledged mutations don't get deleted from a queue — they simply stop being between the head and the newest snapshot after rebase (§4), and the DAG's ref-counting GC reclaims the chunks.

### 1.2 Where it physically lives — and the crash window

Commits are chunks in a content-addressed store keyed by hash, persisted in a single IndexedDB object store named `chunks` (`.reference/mono/packages/replicache/src/kv/idb-store.ts:17`, created at `:216`). But there are **two DAGs**: a per-tab in-memory `memdag` where mutations commit first, and a shared on-disk `perdag`. A background `persist()` copies memdag state into the perdag on a `ProcessScheduler` with a 1 000 ms idle timeout and 500 ms throttle (`.reference/mono/packages/replicache/src/replicache-impl.ts:144-148`, schedulers constructed at `:333-341`, scheduled after writes via `#schedulePersist`, `:1284-1289`).

Honest consequence: **a mutation is durable only after the next persist cycle** (order ~1 s under activity). Kill the tab in that window and the mutation is gone — Replicache's durability is "durable within seconds," not "durable at commit." Any design we write down must state which of those two it promises, because the gap is architecturally load-bearing (per-tab speed vs. cross-tab shared storage is *why* the two-DAG split exists).

## 2. The push protocol: dumb sender, per-client sequencing

The wire mutation is minimal: `{id, name, args, timestamp, clientID}` (`MutationV1`, `.reference/mono/packages/replicache/src/sync/push.ts:36-42`); the request wraps it with `{pushVersion: 1, schemaVersion, profileID, clientGroupID, mutations[]}` (`:58-70`). Push collects **all** pending local commits between the base snapshot and the main head — note: for every client in the group whose commits are on this head, not just the pushing tab's own (`:120-137`; the per-mutation `clientID` is what lets one tab push another tab's mutations, §6). Mutations are sent oldest-first, and the lock is explicitly released before the HTTP call ("Important! Don't hold the lock through an HTTP request!", `:128`).

The client makes **no attempt at exactly-once delivery**. Push is fired by a `ConnectionLoop` — debounced (default `pushDelay = 10` ms, `.reference/mono/packages/replicache/src/replicache-impl.ts:428`), exponential backoff from 30 ms to 60 s on errors (`MIN_DELAY_MS`/`MAX_DELAY_MS`, `.reference/mono/packages/replicache/src/connection-loop.ts:8-9`) — and the same mutation will be re-sent on every push until acknowledged. ## 3. The server contract: `lastMutationID` is the entire dedup/ordering primitive

The server stores exactly one integer per client — `lastMutationID` — and the published contract (in-repo docs = replicache.dev) is explicit:

- **Atomicity**: "The effects of a mutation … and the corresponding update to the `lastMutationID` must be revealed atomically by the datastore … in a SQL database both changes should be committed as part of the same transaction. If [not], the sync protocol will have undefined and likely mysterious behavior." (`.reference/mono/packages/replicache-doc/docs/reference/server-push.md`, "Mutations are Atomic and Ordered" section)
- **Dedup**: "All mutations with `id`s less than or equal to the client's current `lastMutationID` must be ignored." (same file, "Push Launch Checklist")
- **Gap rejection**: "All mutations with `id`s greater than the client's current `lastMutationID+1` must be ignored." (same list) — no out-of-order application, ever.
- **Ack channel**: the ack is *not* the push response — "The server marks … that mutation was applied by returning a `lastMutationID` in the `PullResponse` greater than or equal to its mutation id. Replicache will continue retrying a mutation until the server marks the mutation processed in this way." ("Mutation Status" section). Push responses are advisory; the pull/poke path is the source of truth.

The reference server schema is three fields: `ReplicacheClient = {id, clientGroupID, lastMutationID}` (`.reference/mono/packages/replicache-doc/docs/strategies/reset.md`, Schema section). That's it. Everything an outbox needs from a server is one transactional counter per client — this is the single most transferable fact in this document.

## 4. Rebase on pull: replay intent over the new snapshot

Pull writes the server's new snapshot to a **separate head** (`SYNC_HEAD_NAME`), carrying merged `lastMutationIDs` for the whole client group (`.reference/mono/packages/replicache/src/sync/pull.ts:286-301`). Then `maybeEndPull` (`:304-434`):

1. Collects local commits on the main head whose `mutationID >` the sync head's `lastMutationID` *for that commit's client* (`:360-379`) — i.e., **the new snapshot itself tells the client which pending mutations were consumed**; those are dropped by simply not replaying them.
2. Survivors are replayed oldest-first (`:382`) — each replay re-executes the mutator function against the sync head via `rebaseMutationAndCommit` (`.reference/mono/packages/replicache/src/db/rebase.ts:126-150`), producing *new* local commits with `originalHash` pointing at the originals.
3. Only when zero replays remain does main head := sync head, atomically (`:430-435`). Subscriptions diff against the final state — no intermediate flicker.

Conflict resolution is therefore **rebase-by-re-execution**: the mutator's own code, run against fresh state, is the merge function. Two details worth stealing verbatim:

- **Missing mutator ≠ stuck queue**: if a mutator name no longer exists in code (deploy rolled forward), Replicache stubs a no-op, logs an error, and keeps the queue moving — "the server-side mutation will still get sent. This doesn't remove the queued local mutation, it just removes its visible effects" (`.reference/mono/packages/replicache/src/db/rebase.ts:44-59`).
- **Overlap guard**: `maybeEndPull` verifies the sync snapshot's basis is the current main snapshot and aborts on "Overlapping syncs" (`:350-352`) — concurrent pulls can't interleave rebases.

## 5. Poison pill: the client has NO local escape hatch — unblocking is a server obligation

Nothing in the client can skip a pending mutation. The queue advances only when the server's `lastMutationID` advances. The docs put the burden entirely on the push endpoint, and are blunt about the failure mode: "It is possible to deadlock a client if it pushes a mutation that _always_ causes an error that stops processing. No other mutations from that client can make progress." The prescribed policy (`server-push.md`, Error Handling + Launch Checklist):

- **Permanent error** → "the server must still mark the mutation as processed by updating the `lastMutationID`" — skip it, count it, move on.
- **Temporary error** → abort without updating `lastMutationID`; "Replicache will consider the server offline and try again later." With the caution: "Temporary errors block synchronization … a server should only do this when it definitely will be able to process the mutation later."

Zero automates exactly this policy server-side. CRUD path (`zero-cache` mutagen): on error, re-run the mutation once in **"error mode"** — skip the actual effects, *just* increment `lastMutationID`, and return the error to the app (`.reference/mono/packages/zero-cache/src/services/mutagen/mutagen.ts:196-262`; the comment block at `:202-233` is the best short essay on why "is this an app error or an infra error?" is undecidable and a retry-once-in-error-mode policy is the pragmatic answer). The LMID bump itself is a single upsert `DO UPDATE SET "lastMutationID" = current."lastMutationID" + 1 … RETURNING` inside the same SERIALIZABLE transaction as the write effects (`:447-453`); `received < lmid` throws `MutationAlreadyProcessedError` which is swallowed as a no-op (`:457-462`, `:264-268`), `received > lmid` is an `InvalidPush` protocol error (`:463-467`). Custom-mutator path (`zero-server`, running inside the *user's* Postgres): identical check-and-increment via `updateClientMutationID` in the app transaction (`.reference/mono/packages/zero-server/src/process-mutations.ts:660-683`), with typed `OutOfOrderMutation` / `MutationAlreadyProcessedError` (`:685-694`), and on app error the mutation is re-run with the mutator skipped while an error **result is journaled** for delivery to the client (`:617-627`).

There is one client-side terminal state: if the server answers `ClientStateNotFound` or `VersionNotSupported`, the whole client *group* is marked `disabled` in the perdag — "A disabled client group prevents pulls and pushes from happening" (`.reference/mono/packages/replicache/src/persist/client-groups.ts:231-250`) — and the app is told to reload (`onClientStateNotFound` defaults to `reload`, `.reference/mono/packages/replicache/src/replicache-impl.ts:373-381`). Pending mutations of a disabled group are abandoned. That is the honest bottom of the design: server-side state loss ⇒ client-side queue loss, by construction.

## 6. Multi-tab: no leader election for push — the LMID makes duplicate senders safe

Structure: each tab = one **client** (random 128-bit ID, `.reference/mono/packages/replicache/src/persist/make-client-id.ts:7-10`); all tabs of one app in one browser profile share a **client group** = one perdag + one server-side sync stream. The perdag stores a client map (`ClientV6 = {heartbeatTimestampMs, refreshHashes, persistHash, clientGroupID}`, `.reference/mono/packages/replicache/src/persist/clients.ts:68-96`) and a client-group map whose two mutation-ID records are the cross-tab coordination surface: `mutationIDs` (highest local per client) and `lastServerAckdMutationIDs` (highest server-acked per client), documented as existing precisely so *other* clients can detect unacked pending mutations without loading the commit graph (`.reference/mono/packages/replicache/src/persist/client-groups.ts:29-56`).

Push ownership: **every tab pushes, concurrently, including each other's mutations** (§2). There is no lock and no leader. The Zero mutation tracker states it flatly: "Each tab sends all mutations for the client group … N clients can send the same mutation concurrently" (`.reference/mono/packages/zero-client/src/client/mutation-tracker.ts:341-346`). This only works because the server's `lastMutationID` check makes duplicate delivery a no-op — the dedup primitive *is* the multi-tab strategy. Coordination that does exist is advisory, via `BroadcastChannel`: after a tab persists, it broadcasts `{clientGroupID, clientID}` so sibling tabs schedule a `refresh` from the shared perdag (`.reference/mono/packages/replicache/src/on-persist-channel.ts:24-52`, handled at `.reference/mono/packages/replicache/src/replicache-impl.ts:1291-1297`); a separate channel announces new client groups (`.reference/mono/packages/replicache/src/new-client-channel.ts`).

## 7. Lifecycle and GC: the 24-hour cliff, and mutation recovery as the safety net

- **Heartbeat**: each live client stamps `heartbeatTimestampMs` every 60 s (`.reference/mono/packages/replicache/src/persist/heartbeat.ts:13`).
- **Client GC**: every 5 min, clients inactive > 24 h are deleted from the client map (`CLIENT_MAX_INACTIVE_TIME = 24h`, `GC_INTERVAL = 5min`, `.reference/mono/packages/replicache/src/persist/client-gc.ts:18-23`; the collector never collects the running client, `:68-71`). The constant's own doc comment states the product meaning: this is "the maximum time a tab can be in the background (frozen) and still be able to sync when it comes back."
- **Client-group GC**: groups with no clients are removed — *unless* they still have pending mutations and mutation recovery is enabled (`.reference/mono/packages/replicache/src/persist/client-group-gc.ts:40-71`).
- **Mutation recovery**: the mechanism that makes dead-tab mutations survivable. Every 5 min (`RECOVER_MUTATIONS_INTERVAL_MS`, `.reference/mono/packages/replicache/src/replicache-impl.ts:152`) and on reconnect, a live client walks (a) other client groups in its own perdag and (b) *other IndexedDB databases of the same app* (older format versions included), pushes their unacked mutations on their behalf, and updates their `lastServerAckdMutationIDs` (`.reference/mono/packages/replicache/src/mutation-recovery.ts:91-145`, cross-DB walk at `:115-137`). Recovery that hits `ClientStateNotFound`/`VersionNotSupported` disables that group rather than retrying forever (`:236-260`).

The composed guarantee is subtle and worth stating precisely: pending mutations survive tab death **if some tab of the same app comes online within the GC windows**. A user who goes offline, closes the tab, and returns after the relevant windows have elapsed can lose queued writes — the system is explicitly not designed as an unbounded store-and-forward queue.

## 8. Zero's evolution: same engine, three deliberate reversals

Zero (`zero-client`) embeds Replicache as its local layer — custom mutators are literally registered as Replicache mutators (`.reference/mono/packages/zero-client/src/client/make-replicache-mutators.ts:22-63`), so the DAG log, persist cycle, rebase, and recovery of §§1–7 all still exist, with `kvStore` defaulting to `'idb'` (falling back to `'mem'` without IndexedDB — `.reference/mono/packages/zero-client/src/client/zero.ts:498-507`). What changed:

1. **CRUD → custom mutators (speculative client / authoritative server).** A custom mutator runs immediately client-side (optimistic, against the local ZQL store) and is *re-run authoritatively on the user's own API server* inside the app's Postgres transaction, LMID check included (§5). The client's optimistic version is discarded by ordinary rebase once the authoritative result arrives. CRUD mutators — where zero-cache itself applied writes — are deprecated, and running both concurrently is so racy that mutagen has a special retry for the resulting out-of-order errors (`.reference/mono/packages/zero-cache/src/services/mutagen/mutagen.ts:269-288`).
2. **Push became a WebSocket stream with a sender-side dedup optimization.** Zero's pusher sends each mutation as its own `push` message and skips mutations already sent on this connection via `#lastMutationIDSent` — except recovery pushes, which always send everything (`.reference/mono/packages/zero-client/src/client/zero.ts:1921-1990`). Per-mutation results come back and are correlated by a `MutationTracker`: ephemeral ID → `{client, server}` promise pair, resolved either by an explicit push result or *implicitly* by any observed LMID advance (pokes / the `connected` handshake carrying `lastMutationID`) (`.reference/mono/packages/zero-client/src/client/mutation-tracker.ts:104-119, 266-310`). Server-side, one `PusherService` per client group queues and forwards to the user's API server, keeping per-client order while interleaving clients (`.reference/mono/packages/zero-cache/src/services/mutagen/pusher.ts:55-96`), and journals mutation *responses* for redelivery until the client acks them (`ackMutationResponses`/`deleteClientMutations`, `:116-124`, `:47-50`).
3. **Offline writes were turned OFF.** The `MutatorProxy` subscribes to connection state; in `Disconnected`/`Error`/`Closed` it (a) rejects **all outstanding** server promises and (b) short-circuits *new* mutations to a rejection without running the local mutator at all (`.reference/mono/packages/zero-client/src/client/mutator-proxy.ts:73-121`). The docs match the code: "Reads from synced data continue to work, but writes are rejected … you should prevent the user from inputting data to your application to avoid data loss"; writes queue only during the `connecting` grace state (~1 min, configurable `disconnectTimeoutMs`); "Zero is not designed for long periods offline"; offline write support is something they'd "like to revisit … not a priority right now" (https://zero.rocicorp.dev/docs/offline, fetched 2025-11-04).

Reversal 3 is the strategic datum. The team with the most browser-outbox production experience concluded that *silent* queuing of writes that may fail authoritative re-execution much later is worse UX than telling the user they're offline — because with server-authoritative custom mutators (their reversal 1), an offline queue is a queue of **unvalidated promises**, and hours-later rejections are unresolvable UX. Replicache could afford hours-offline queues because its mutators were the same code both sides and the docs pushed conflict pain onto app-level history/undo ("the potential for serious conflicts grows the longer users are disconnected … we recommend implementing a concept of history so that users can undo merges", `.reference/mono/packages/replicache-doc/docs/concepts/offline.md`). Replicache's honest offline claim, for the record: "A tab can go offline and continue to operate for hours to days, then sync up smoothly" — days, bounded by §7's GC cliffs and by the server's ability to compute a pull patch from an arbitrarily old cookie (strategy-dependent; the row-version strategy handles any gap via per-client CVR state, the reset strategy by resending everything — `.reference/mono/packages/replicache-doc/docs/strategies/`).

## 9. The ten mechanisms an outbox MUST have, per this school

Each is load-bearing in the source; skipping any one produces a concrete, named failure.

1. **Store intent, not effects** — mutation = `{name, argsJSON}` replayable against new state (`db/commit.ts:257-267`). Without it: no rebase, only blind write-set overwrite.
2. **Per-client dense sequence numbers** assigned at local commit time, gap-free (`db/rebase.ts:64-72`). Without it: the server cannot dedup or order cheaply.
3. **One transactional `lastMutationID` per client on the server, committed atomically with the mutation's effects** (`server-push.md` atomicity section; live SQL at `mutagen.ts:447-453`). This single integer is the entire exactly-once story. Without it: duplicates or lost writes under retry.
4. **Skip-≤, reject-gap server rule** (`server-push.md` launch checklist; typed errors at `process-mutations.ts:669-694`). Makes at-least-once client delivery safe and out-of-order delivery impossible.
5. **Ack via authoritative state flow, not the push RPC** — the pull/poke's `lastMutationIDChanges` is what clears the queue (`sync/pull.ts:286-301`; `server-push.md` "Mutation Status"). Without it: a lost push response strands or double-clears the queue.
6. **Rebase = drop-consumed + re-execute survivors over the new snapshot on a separate head, swap atomically** (`sync/pull.ts:304-434`). Without it: optimistic state and server state can't reconcile without flicker or double-apply.
7. **A poison-pill policy with teeth**: permanent app error ⇒ server advances LMID anyway (error-mode re-run, `mutagen.ts:196-262`); temporary error ⇒ halt without advancing; missing mutator ⇒ client no-op stub keeps replaying (`db/rebase.ts:44-59`). Without it: one bad mutation deadlocks the client forever — their word ("deadlock", `server-push.md`).
8. **Multi-writer safety by idempotence, not election** — every tab pushes everything; the LMID makes duplicates no-ops; BroadcastChannel refresh is advisory only (`mutation-tracker.ts:341-346`, `on-persist-channel.ts:24-52`). Without it: leader-election complexity or duplicate application.
9. **Lifecycle GC plus a recovery sweep** — heartbeats, 24 h client GC, group GC that spares pending-mutation groups, and periodic recovery pushing dead clients' mutations (`client-gc.ts:18-23`, `client-group-gc.ts:40-71`, `mutation-recovery.ts:91-145`). Without it: unbounded storage growth *or* silent loss of dead tabs' writes — this pair is how you get neither.
10. **An explicit terminal state** — server-lost client state ⇒ disable the group, surface it (`onClientStateNotFound` → reload), abandon its queue (`client-groups.ts:231-250`). Without it: infinite retry against a server that has forgotten you.

And one meta-mechanism that is a product decision, not code: **decide, out loud, whether offline writes are promises you can keep.** Replicache said "hours to days, bring your own undo"; Zero, with server-authoritative mutators, said "no — reject writes while disconnected rather than queue unvalidated intent" (`mutator-proxy.ts:84-121`; zero.rocicorp.dev/docs/offline). An outbox design that doesn't state its position on this axis has not made the hardest choice in the space.
