A mutation is a function that **writes** data. It is the only kind of function allowed to —
[queries](/docs/core-concepts/queries) can only read, and [actions](/docs/core-concepts/actions)
have no `ctx.db` at all. Every write in a stackbase app, no matter how it was triggered, happens
inside a mutation.

## Defining a mutation

A mutation takes a context (`ctx`) and, optionally, arguments, and returns a value:

```ts title="convex/messages.ts"
import { v } from "@stackbase/values";
import { mutation } from "./_generated/server";

export const send = mutation({
  args: { author: v.string(), body: v.string() },
  returns: v.id("messages"),
  handler: (ctx, args) => ctx.db.insert("messages", { author: args.author, body: args.body }),
});
```

- **`args`** validates the mutation's arguments before the handler ever runs. A mismatch — wrong
  type, missing required field, or an extra field — is rejected and the handler never executes.
- **`returns`** is optional, but it's what lets a typed [optimistic
  update](/docs/client/optimistic-updates) know the exact shape of what a mutation resolves to.
- **`handler`** does the writing through `ctx.db`.

## Writing data

`ctx.db` in a mutation is a writer: everything a query's reader can do (`get`, `query(...).collect()`,
`.paginate()` — see [Queries](/docs/core-concepts/queries)), plus three write methods:

- **`ctx.db.insert(table, value)`** — inserts a new document and returns its `_id`.
- **`ctx.db.replace(id, value)`** — replaces a document's fields entirely, given its `_id`.
- **`ctx.db.delete(id)`** — removes a document by its `_id`.

There is no partial-patch method — `replace` takes the *whole* document. To change one field, read
the document first and write back every field, changed or not:

```ts title="convex/items.ts"
export const toggle = mutation({
  args: { id: v.id("items"), done: v.boolean() },
  handler: async (ctx, { id, done }) => {
    const doc = await ctx.db.get(id);
    if (doc === null) return;
    await ctx.db.replace(id, { listId: doc.listId, label: doc.label, done });
  },
});
```

## One serializable transaction

A mutation's entire handler runs as a single **serializable transaction**. Every read it does sees
a consistent snapshot, every write it makes is invisible to everyone else until the handler returns
successfully, and then all of it commits together — or, if the handler throws, none of it does.
There is no partial write: a mutation that inserts three rows and then throws leaves zero rows
behind.

This is also what gives the engine a single, well-defined moment to hook reactivity into: the
commit. There's no ambiguity about "when did this write happen" for the engine to reason about,
because a mutation either fully happened at its commit or didn't happen at all.

## Mutations are deterministic, like queries

A mutation runs under the same purity rule as a query: no `fetch`, no `Date.now()`, no
`Math.random()`. If the engine ever needs to re-run a mutation's handler — for example, replaying it
after an optimistic-concurrency conflict — it needs the same code, given the same arguments and the
same data, to do the exact same writes every time.

When a mutation genuinely needs the current time or a random value, use `ctx.now()` and
`ctx.random()` instead of the global `Date.now()`/`Math.random()` — both are fixed for the life of
that transaction (and any replay of it), rather than drifting between attempts. Anything that
can't be made deterministic this way — a network call, a real UUID, a non-replayable clock read —
belongs in an [action](/docs/core-concepts/actions) instead.

## The commit produces the write-set

When a mutation commits, the engine records exactly which rows it inserted, replaced, or deleted —
its **write-set**. That write-set is then intersected against every subscribed query's recorded
read-set: a query refreshes only if a mutation actually touched data it read. This intersection is
the entire reactive engine — see [How it works](/docs/get-started/how-it-works) and
[Reactivity](/docs/core-concepts/reactivity) for the full mechanism.

## Calling a mutation from a client

On the client, `useMutation` returns a callback bound to a typed function reference:

```tsx
import { useMutation } from "@stackbase/client/react";
import { api } from "../convex/_generated/api";

function Composer({ author }: { author: string }) {
  const send = useMutation(api.messages.send);

  return (
    <button onClick={() => void send({ author, body: "hello" })}>
      Send
    </button>
  );
}
```

Calling it sends the mutation over the sync connection and resolves once it commits. The write's
effects show up wherever a live query's read-set overlaps it — including in the very same
component, if it's subscribed with `useQuery`. The full client surface, including
`withOptimisticUpdate` for instant local feedback, is in [Client SDK](/docs/client/client-sdk) and
[Optimistic updates](/docs/client/optimistic-updates).
