{/* diataxis: tutorial */}

You'll build a tiny backend with a `messages` table, a query that lists messages, and a mutation that
adds one. Then you'll watch the list update the moment you write a new row. It takes a few minutes,
and every step below is real.

<Callout title="Before you start">
  You need **Bun** or **Node.js 22.5+** and a package manager. Your backend lives in a folder called
  `convex/`. That's just the default name (everything you import is `@stackbase/*`), and you can
  change it later with `--dir`.
</Callout>

<Steps>

<Step>

### Install stackbase

<Tabs items={['npm', 'pnpm', 'bun']}>

```bash tab="npm"
mkdir my-app && cd my-app && npm init -y
npm install @stackbase/values @stackbase/client
npm install -D @stackbase/cli
```

```bash tab="pnpm"
mkdir my-app && cd my-app && pnpm init
pnpm add @stackbase/values @stackbase/client
pnpm add -D @stackbase/cli
```

```bash tab="bun"
mkdir my-app && cd my-app && bun init -y
bun add @stackbase/values @stackbase/client
bun add -d @stackbase/cli
```

</Tabs>

`values` gives you the schema and validators, `client` is the browser and Node client, and `cli` is
the `stackbase` command you'll run.

</Step>

<Step>

### Define a schema

Create `convex/schema.ts`. It lists your tables and the shape of each row:

```ts title="convex/schema.ts"
import { defineSchema, defineTable, v } from "@stackbase/values";

export default defineSchema({
  messages: defineTable({
    author: v.string(),
    body: v.string(),
  }),
});
```

Every row also gets an `_id` and a `_creationTime` automatically, so you never declare those. The full
list of field types (`v.number()`, `v.id(...)`, `v.optional(...)`, and so on) is in
[Schema & tables](/docs/core-concepts/schema-and-tables).

</Step>

<Step>

### Write a query and a mutation

Create `convex/messages.ts`:

```ts title="convex/messages.ts"
import { v } from "@stackbase/values";
import { query, mutation } from "./_generated/server";

// Reads data. Your app subscribes to this and gets live updates.
export const list = query({
  handler: (ctx) => ctx.db.query("messages", "by_creation").collect(),
});

// Writes data. Runs as one transaction.
export const send = mutation({
  args: { author: v.string(), body: v.string() },
  handler: (ctx, args) => ctx.db.insert("messages", args),
});
```

`by_creation` is a built-in index every table has, so `list` returns every message oldest-first with
no index of your own.

<Callout type="info">
  Your editor will flag `./_generated/server` as missing. That's expected. The next step creates it.
</Callout>

</Step>

<Step>

### Run it

```bash
npx stackbase dev
```

`stackbase dev` generates the typed `convex/_generated/` helpers, starts the engine on a local SQLite
file, serves a dashboard, and reloads whenever you edit a file. It prints its URL and a dev admin key:

```
stackbase dev  →  http://127.0.0.1:3000   (dashboard: /_dashboard)
admin key      →  sk_dev_3f8a1c...
```

</Step>

<Step>

### Watch it update live

You don't even need client code to see reactivity. The dashboard's data browser is itself a live
subscription:

1. Open the dashboard URL that `stackbase dev` printed.
2. Open the **data browser** and select the `messages` table. Leave it on screen.
3. Open the **function runner** and call `messages:send` with `{ "author": "Ada", "body": "hello" }`.
4. The new row shows up in the data browser right away, with no refresh.

That live view is a subscription to `messages:list`, the same thing a real app holds. A mutation wrote
a row, and every subscription reading that table refreshed on its own. You wrote no polling and no
invalidation code.

</Step>

</Steps>

## Connect a client

With the server running, point a client at the `ws://127.0.0.1:3000/api/sync` endpoint it opened:

<Tabs items={['React', 'Plain JS']}>

<Tab value="React">

```tsx title="App.tsx"
import { StackbaseClient, webSocketTransport } from "@stackbase/client";
import { StackbaseProvider, useQuery, useMutation } from "@stackbase/client/react";
import { api } from "./convex/_generated/server";

const client = new StackbaseClient(webSocketTransport("ws://127.0.0.1:3000/api/sync"));

function Messages() {
  const messages = useQuery(api.messages.list, {});
  const send = useMutation(api.messages.send);
  if (messages === undefined) return <p>Loading…</p>;
  return <ul>{messages.map((m) => <li key={m._id}>{m.author}: {m.body}</li>)}</ul>;
}

export function App() {
  return (
    <StackbaseProvider client={client}>
      <Messages />
    </StackbaseProvider>
  );
}
```

`useQuery` subscribes the component to the query. It returns `undefined` until the first result
arrives, then re-renders on its own every time the result changes. That's the whole pattern.

</Tab>

<Tab value="Plain JS">

```ts title="client.ts"
import { StackbaseClient, webSocketTransport } from "@stackbase/client";
import { api } from "./convex/_generated/server";

const client = new StackbaseClient(webSocketTransport("ws://127.0.0.1:3000/api/sync"));

// Subscribe. The callback fires again every time the result changes.
client.subscribe(api.messages.list, {}, (messages) => {
  console.log("messages:", messages);
});

// Write. The subscription above fires right after this commits.
await client.mutation(api.messages.send, { author: "Ada", body: "hello" });
```

</Tab>

</Tabs>

See [Client SDK](/docs/client/client-sdk) for the full hook surface, and
[Tutorial](/docs/get-started/tutorial) for a complete app built this way.

## Going deeper

<Accordions type="single">

<Accordion title="stackbase dev flags">

| Flag | Default | What it does |
|---|---|---|
| `--port` | `3000` | HTTP and WebSocket port |
| `--ip` | `127.0.0.1` | Bind address |
| `--dir` | `convex` | Function directory |
| `--data` | `.stackbase/data.db` | SQLite file path |
| `--web` | (none) | Serve a built web UI alongside the API |
| `--database-url` | (SQLite) | Use Postgres instead |
| `--storage-bucket` | (local disk) | Use S3-compatible file storage |

Run `stackbase help` for every command and flag.

</Accordion>

<Accordion title="What's in convex/_generated/">

After the first run, `convex/_generated/` holds typed helpers derived from your schema and functions:

- `Doc<"messages">` and `Id<"messages">`: the type of a row, and a typed id to reference one.
- `api`: a typed object mirroring your functions (`api.messages.list`, `api.messages.send`). Pass
  these to `useQuery`/`useMutation` and the client infers the argument and return types.

`stackbase dev` regenerates these on every change, so you rarely run codegen by hand. You do need to
commit `_generated/` before a `stackbase serve` or `stackbase build` deploy, since those don't run
codegen at boot.

</Accordion>

<Accordion title="Where does the data live?">

By default, everything is a single SQLite file at `.stackbase/data.db` (change it with `--data`).
Nothing to install. To use Postgres instead, pass `--database-url postgres://...` and change nothing
in your app. See [Postgres](/docs/deploy/postgres).

</Accordion>

</Accordions>
