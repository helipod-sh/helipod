The framework-agnostic client and React hooks that subscribe to reactive queries and call functions.
This page uses the `examples/chat` app as its running example.

## Construct the client

`StackbaseClient` takes a **transport** — the thing that actually carries the sync protocol.
Use `webSocketTransport(url)` for a real deployment (talking to `stackbase dev`/`stackbase serve`
over the network):

```tsx title="examples/chat/web/main.tsx"
import { StackbaseClient, webSocketTransport } from "@stackbase/client";

const wsProtocol = location.protocol === "https:" ? "wss" : "ws";
const client = new StackbaseClient(webSocketTransport(`${wsProtocol}://${location.host}/api/sync`));
```

The client logic itself is transport-agnostic — an embedded/in-process connection uses
`loopbackTransport` instead, with the same `StackbaseClient` on top. The WebSocket transport
reconnects automatically by default (exponential backoff with jitter); pass
`{ reconnect: false }` to restore a terminal-on-close transport.

## Wrap your app in the provider

The React hooks read the client from context. Wrap your app once, near the root:

```tsx title="examples/chat/web/main.tsx"
import { StackbaseProvider } from "@stackbase/client/react";

createRoot(root).render(
  <StackbaseProvider client={client}>
    <Chat />
  </StackbaseProvider>,
);
```

Everything below assumes a component is rendered under a `StackbaseProvider`.

## Subscribe with `useQuery`

`useQuery` takes a generated function reference (`api.messages.list`) and its arguments, and
returns the current result — `undefined` until the first result arrives:

```tsx title="examples/chat/web/main.tsx"
import { useQuery } from "@stackbase/client/react";
import type { Api } from "../convex/_generated/api";
import { anyApi } from "@stackbase/client";

const api = anyApi as Api;

function Chat() {
  const messages = useQuery(api.messages.list, { conversationId: CONVERSATION_ID });
  if (messages === undefined) return <p>connecting…</p>;
  return (
    <ul>
      {messages.map((m) => (
        <li key={m._id}>{m.author}: {m.body}</li>
      ))}
    </ul>
  );
}
```

Passing a codegen `api.*` reference infers typed args and a typed return value from the query's
declared `args`/`returns` validators. Under the hood, `useQuery` calls `client.subscribe(...)` in
an effect and re-subscribes only when the reference or its (serialized) arguments change; the
subscription is torn down automatically when the component unmounts.

There is no manual refresh, no polling interval, and nothing to invalidate by hand — the component
re-renders on its own whenever the query's read set is touched by a committed write. See
[Queries](/docs/core-concepts/queries) and [How it works](/docs/get-started/how-it-works) for why
that works.

## Call functions with `useMutation` and `useAction`

`useMutation` returns a callable that runs a mutation and resolves with its return value:

```tsx title="examples/chat/web/main.tsx"
import { useMutation } from "@stackbase/client/react";

function Chat() {
  const send = useMutation(api.messages.send);

  function submit(body: string) {
    void send({ conversationId: CONVERSATION_ID, author, body });
  }
  // ...
}
```

`useAction` mirrors it for actions — the side-effecting functions that run outside the transaction
(`fetch`, timers, etc.):

```tsx
import { useAction } from "@stackbase/client/react";

const sendWelcomeEmail = useAction(api.users.sendWelcomeEmail);
await sendWelcomeEmail({ userId });
```

Unlike `useQuery`, neither hook subscribes to anything — calling the returned function runs the
mutation/action once and returns a promise. `useMutation(ref)` also has a `.withOptimisticUpdate(fn)`
chain for rendering a predicted result before the round trip completes — see
[Optimistic updates](/docs/client/optimistic-updates).

## How a query updates automatically

There is no code in the component that says "refresh when a message is sent." The mechanism is the
same one described in [How it works](/docs/get-started/how-it-works): while `messages.list` runs,
the engine records exactly which rows it read; when `messages.send` commits, the engine computes
what it wrote and pushes a new result to every subscription whose read set the write touches. The
client's job is just to apply that push and re-render — which is exactly what `useQuery` does for
you.

Open the chat example in two tabs and send a message in one: the other tab's `messages` list
updates with no refresh, because both tabs are subscribed to the same query and the mutation's
write intersects both of their read sets.
